Source
Expand description

Array-side sources.

We categorize two types of sources:

  • Opaque: Not in-memory (by some definition). These have chunk / shard preferences like Zarr, that are taken into account at planning time, and reads that can fail (I/O).
  • Transparent: generate any requested brick on demand, cheaply. These sources are infallible: any error should have been caught at plan-time. A Source that can fail at read time must be an OpaqueSource. Example: In-memory arrays (InMemorySource) and functional arrays defined by an equation rather than stored data. Examples today are FillSource today; Later we may have arange / eye / date_range etc. No inherent chunk grid; lowering assigns one freely.

Both are traits; lowering wraps each in a grid-carrying adapter (LoweredTransparent / LoweredOpaque) that implements [brix::Source].

Structs§

FillSource
Constant functional array: shape filled with value. Emits materialized dense bricks — the runtime carries no constant payload kind; all-fill regions are an iteration-space (occupancy) concern.
InMemorySource
In-memory data as a transparent source. Wraps the payload enum — not DenseArray — so one impl covers Dense and Extension (and future Sparse) data: post-Fill-deletion, ArrayPayload is “in-memory array data of any kind”. The entry point for satellite extension payloads.
LoweredLiteral
The lowered form of crate::expr::ArrayOp::Literal. TODO: I am unsure if modeling this as a source makes sense but it’s cute. we emit 0-rank arrays instead of scalars. We could add ArrayPayload::Scalar instead.
LoweredOpaque
Wraps an opaque user-provided source with the grid lowering chose. inner.read is the user-facing element-space read; this adapter is the brix::Source impl.
LoweredTransparent
Wraps a transparent source with the grid lowering chose — the single brix::Source impl for every transparent source, mirroring LoweredOpaque. Cost is numel × itemsize from grid + dtype (extension dtypes supply their own itemsize).
SourceNativeChunkGrid
The chunk grid an OpaqueSource owns natively — the on-disk / in- storage chunking the source was constructed with (Zarr chunks, block-sparse occupancy-carrying grids, etc.). Distinct, non-substitutable type so it can’t be confused with a plain ChunkGrid that the planner picked or one that flows through LoweredOpaque at runtime. Constructed explicitly via SourceNativeChunkGrid(grid) or grid.into(). Read access is transparent via Deref<Target=ChunkGrid>; mutation requires going through .0 deliberately.

Traits§

OpaqueSource
Opaque source — I/O-backed (Zarr, Arrow, etc.). The optimizer can’t see through this; it only knows shape, layout, and ArrayPayloadKind. The source owns its chunk grid (data is already chunked on disk).
TransparentSource
Transparent source — generates any requested brick on demand, cheaply and infallibly. The trait behind functional arrays: an impl is a generation rule (constant, range, function) plus its shape and dtype.
