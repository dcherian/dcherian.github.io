Expand description

A schedule is a tree of nested loops with computation steps placed at specific loop levels. The executor interprets it — making no decisions except how/whether to parallelize (if allowed); prefetching of sources, etc.

Every compute step (Init/Apply/Write) carries a precomputed AccessRelation — the translation from this band’s iteration space to the producer’s iteration space. The planner computes it once via producer_grid.layout.compose(view) at schedule emission. The executor evaluates it against current loop indices to get the brick coord.

Today access is identity-name-keyed (iter_space is built directly from the producer grid), but the slot exists so future iter_space rewrites (strip mining, slice-pushdown, distributed sub-bands) don’t require schema changes.

The three-phase pattern (Init/Apply/Finalize) is driven by core dims:

  • Nodes with core dims get all three phases placed at different loop levels.
  • Nodes without core dims get only Apply.

§

Domain

Establish the iteration space for the enclosed body. Loop nodes inside look up their axis extents from here. One Domain per band; multi-band plans nest or sequence several. A distributed executor may construct Schedules for each worker by simply rewriting these Domain nodes to the appropriate subspace the worker is responsible for.

Loop

Iterate over a chunk dimension, identified by axis name. The range is looked up from the enclosing Domain at execute time. The enclosing Domain’s iteration space may be nested (e.g. for coarsen/rolling) So we include a mode_index to grab the “i-th mode” or “i-th sublayout” (Czeka 2026 Section 2.4.1) of the enclosing Domain’s iter_space.

Zip

Co-iterate axes over an explicit coordinate table (soon to be) filtered by the overarching Domain range (similar to Loop). coords[i] is the i-th present coord, length axes.len(), values in axes positional order.

Sequence(Vec<Schedule>)

Execute a list of schedule steps in specified order at this loop level. Used to express top-level sequences (e.g. InitSink, loop nest, FinalizeSink).

§

Init

Execute a list of schedule steps in any order at this loop level. Unused today Initialize state for a compute node that has core dimensions. Placed before the core-dim loops begin.

Example, for Reduce, this creates accumulator state sized to the output brick.

access is this node’s iter→global access relation — used to derive the output coord that sizes the accumulator’s initial state.

Fields

§node: NodeName
§access: AccessRelation
§

Apply

Per-iteration “compute” step. Examples:

  • Source: read brick at current coord into workspace.
  • Blockwise: read inputs from workspace, broadcast, apply op, store result.
  • Reduce: fold workspace[input] into accumulator.

inputs names the workspace entries this step reads, in port order. Each input brick’s coordinate was already fixed by the producer’s own Apply (recompute model: the emission immediately precedes this read), so the edge carries no access of its own. The planner controls placement — a source Apply can be hoisted above inner loops for reuse when broadcasting is involved.

access is this node’s own iter→global translation (e.g. for the accumulator result).

Sinks never appear in Apply — they have their own InitSink / WriteSink / FinalizeSink variants.

Fields

§node: NodeName
§inputs: Vec<NodeName>
§access: AccessRelation
§

Finalize

Extract result from a compute node’s state after core-dim iteration completes. Placed after the core-dim loops exit. Produces a brick in workspace for routing to the next step.

For Reduce: calls accumulator.finalize() → workspace brick.

§node: NodeName

WriteSink

Write workspace[node] into the sink at the current brick coord, and release workspace[node]. This is the explicit memory-release point for a brick on its way out of the system.

access is this node’s iter→global translation — used to derive the brick coord at which to write into the sink.

Fields

§node: NodeName
§sink: NodeName
§access: AccessRelation
§

FinalizeSink

Finalize a sink (commit writes, close files, etc.). Placed once, after every WriteSink into that sink has executed.

Fields

§sink: NodeName
