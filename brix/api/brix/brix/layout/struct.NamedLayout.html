Expand description

A Layout with an axis name per top-level mode.

Layout is deliberately positional (CuTe convention): shape and stride are unnamed IntTuples, which is what makes the layout algebra (composition, tiling, complement) work without dragging names through every transform.

Scheduling, though, needs to align axes by name — which iter axis corresponds to which producer axis, which stateful’s core dim is which.

NamedLayout is the wrapper that pairs them: it owns the naming policy and will eventually own the name-preserving versions of layout transforms (tile, split, fuse). For now it’s a plain pair — methods will accrete as the refactor lands.

We also introduce offsets here. In CuTe, the offset is stored on a Tensor not the Layout.

Invariant: names.len() == layout.rank() (one name per top-level mode). Hierarchical splitting under tiling (e.g. i → (i_outer, i_inner)) is deferred — the rule for how names split is a policy choice tied to whichever rewrite introduces the hierarchy.

pub fn broadcast_with( &self, other: &NamedLayout, ) -> Result<NamedLayout, LayoutError>

Unify two iteration spaces by axis name, checking n_chunks agree (with size-1 broadcasting).

Per-axis rules:

  • Name appears in only one side → keep it.
  • Name in both, same n_chunks → kept as-is.
  • Name in both, one side is n_chunks == 1 → the 1 side broadcasts; the non-1 side wins.
  • Name in both, different non-1 extents → LayoutMergeError::ExtentMismatch.
  • Name in both, same non-1 extent but different non-zero offsets → LayoutMergeError::OffsetMismatch (the two layouts see different regions of the producer; this merger can’t represent that).

Per-chunk compatibility (same name, same extent, different chunk partitions) isn’t the concern here — upstream (ChunkGrid::new and broadcast_grids) keeps the “same name + same extent ⇒ same chunks” invariant across a plan.

Result preserves self’s axis order, with other’s new axes appended. Order is merge-deterministic, not semantic — callers treat the result as an unordered set.

pub fn extend( &self, axes: impl IntoIterator<Item = (AxisName, usize)>, ) -> Result<NamedLayout, LayoutError>

New NamedLayout with extra axes appended (each as (name, extent), offset 0). Strides are rebuilt row-major via native_layout, matching from_shape / broadcast_with.

pub fn ignoring_modes(&self, ignored: &[(AxisName, usize)]) -> NamedLayout

Identity-tiler view over this iteration layout — the per-edge view the planner stamps and composes against a producer to translate iteration coords into producer chunk coords.

Per top-level axis: flat n → n : 1; tiled (n_out, k) → (n_out, k) : (k, 1), so producer ∘ view reads c·k + j on a coarsened axis (broadcast on absent axes falls out of compose’s name match). Names are preserved.

Pass stride 0 on (axis, mode_index) pairs the access must be invariant under (same shape Schedule::Loop uses). Use case today: a Reduce’s Apply own-output access must be invariant under each fold-tile Loop the planner just placed, so the accumulator address doesn’t move as the fold sweeps. Pass &[] for the plain identity-tiler.

Source

pub fn compose(&self, view: &NamedLayout) -> Layout

Compose this layout (a flat producer chunk grid) with a per-edge view: self ∘ view, name-aware. The result maps the view’s iteration coords to this layout’s linear brick index — shapes come from view (the new domain), strides from self (the codomain it addresses).

For each top-level axis of view, the producer’s (count, stride) is matched by name — size-1 / absent collapses to the broadcast atom (1 : 0) (stepping it re-reads the same brick) — then composed with the view’s mode for that axis via Layout::compose. Transpose falls out of name matching; broadcast from the (1:0) atom; coarsen / projection from the view’s mode ((n_out,k):(k,1) reads c·k + j; (n_out,k):(1,0) projects to c).

The per-edge view (read or projection) is stamped by the planner from the iteration space — see native_layout / Self::ignoring_modes — so the projection-vs-read choice lives there, not in stride arithmetic re-derived from producer counts.

Source

pub fn logical_divide( &self, axis: &AxisName, tile: usize, ) -> LayoutResult<NamedLayout>

Split the named axis by an integer tile, in place, keeping the axis name and offset. The top-level mode axis : n becomes the hierarchical axis : (ceil(n/tile), tile) via row-major Layout::logical_divide — depth 0 is the surviving rest mode (keeps the name), depth 1 the folded tile mode. names/offsets are unchanged (still one entry per top-level mode); the inner Layout’s shape tuple carries the per-depth extents.

Errors with LayoutError::UnknownAxis if axis is absent, or LayoutError::Unsupported if the mode is already tiled (re-splitting a hierarchical mode — i.e. nested tiling — is deferred).

Source

pub fn mode(&self, axis: &AxisName) -> LayoutResult<Layout>

The named top-level mode as its own Layout — name resolution over CuTe positional Layout::get. Index into it with get(i) for sub-modes (e.g. get(1) is the tile mode of a Self::logical_divided axis), size() for extents, rank() for the sub-mode count (the valid mode_index range: 1 flat, 2 after one divide).

Source

pub(crate) fn eval_axis( &self, axis: &AxisName, indices: &NamedCoord, ) -> LayoutResult<usize>

Evaluate the inner Layout from a NamedCoord (iteration indices keyed by (axis, mode_index), where mode_index is the CuTe get<I> position within the axis’s mode).

For each top-level mode, assembles its natural coord as an IntTuple congruent to that mode (each leaf drawn from (axis, position), get<0>, get<1>, …), then evaluates via Layout::crd2idx — the hierarchical coord is fed straight through, no linearization. A flat (untiled) axis is a single position-0 leaf, so this also covers the flat case.

Returns LayoutError::UnknownAxis (keyed "axis@pos") if any required (axis, mode_index) is missing from indices.
