Source
Expand description

Array-side sources.

We categorize two types of sources:

  • Opaque: Not in-memory (by some definition). These have chunk / shard preferences like Zarr, that are taken in to account at planning time.
  • Transparent: These are “in-memory” (by some definition). Thus a ndarray is a Transparent::Dense source. Other transparent sources are those that can be generated given a ChunkGrid and a BrickCoordinate. E.g. TransparentSource::Fill models arrays that are just a constant broadcast out to the appropriate shape. Other “parameteric” arrays may be modeled in the future: e.g. those constructed from the equivalents of np.repeat or np.tile. At the moment there is a partner variant in ArrayPayloadKind that allows downstream pattern matching.

By implication, Opaque sources are more expensive than Transparent sources. Transparent sources are enums because we expect to introduce optimizations by pattern-matching on the specific kind. OpqueSource is a trait because I don’t expect to pattern-match (yet).

LoweredOpaque

SourceNativeChunkGrid
The chunk grid an OpaqueSource owns natively — the on-disk / in- storage chunking the source was constructed with (Zarr chunks, block-sparse occupancy-carrying grids, etc.). Distinct, non-substitutable type so it can’t be confused with a plain ChunkGrid that the planner picked or one that flows through LoweredOpaque at runtime. Constructed explicitly via SourceNativeChunkGrid(grid) or grid.into(). Read access is transparent via Deref<Target=ChunkGrid>; mutation requires going through .0 deliberately.

Enums§

TransparentSource
Transparent sources — optimizer can pattern-match on the variant. No inherent chunk grid; the planner (via lowering) assigns one freely.

Traits§

OpaqueSource
Opaque source — I/O-backed (Zarr, Arrow, etc.). The optimizer can’t see through this; it only knows shape, layout, and ArrayPayloadKind. The source owns its chunk grid (data is already chunked on disk).
