The Brix Book

Algorithms

How the structural ops in the pipeline visualizer are implemented. Every one follows the same three-part shape:

  1. Plan-time chunk-grid math (array/lower.rs + array/chunk_grid/*): construct the output ChunkGrid and the chunk-coord routing from output chunks back to input chunks.
  2. DAG embedding: lowering emits a mid-IR Expr node whose Input edges each carry access: HashMap<AxisName, AxisAccess>. An axis is either AxisAccess::Composed(Frame) — a structural transform the planner folds into the loop nest (NamedLayout::compose / logical_divide) — or AxisAccess::Translated(TranslatedAccess) — a per-coord override applied at runtime by AccessRelation::resolve_coord after the affine pass. Absence from the map is identity (pass-through / broadcast).
  3. Within-brick element math: the concrete Kernel / Accumulator bakes in both the input and output ChunkGrid plus the original indexer/view, so it can derive element-space bounds locally from one brick coordinate.

At a glance (visualizer legend token → actual variant):

Slicing

ArrayOp::Slice(HashMap<AxisName, Slice>) lowers to Op::Blockwise(SliceKernel). Three pieces of information construct the output:

  • Output chunk grid — plan time, in chunk_grid::select::output_grid_and_offsets. Per sliced axis, each output chunk is the intersection of one source chunk with [start, end) step step, so the per-chunk sizes are encoded as AxisChunks::RLE. The output stays AxisChunks::Uniform only on the aligned fast path (step 1, start/end on chunk boundaries); slicing an AxisChunks::Variable axis is a hard error.
  • Per-axis chunk offset on the edge — output_grid_and_offsets also returns k0 per axis: the source chunk that output chunk 0 maps to. Lowering embeds it on the consumer edge as AxisAccess::Translated(TranslatedAccess::Independent(AxisOverride::Offset(k0))) for every input axis (Offset(0) is a no-op for untouched axes). At runtime AccessRelation::resolve_coord adds the offset to the affine-resolved producer coord. The override stack is ordered: two slices separated by a Blockwise (so fuse_adjacent_selects can’t collapse them) accumulate as stacked entries applied in planner descent order — Offset(1) then Offset(2) reads the producer at chunk c + 3 (the “stacked slices” workload).
  • Within-brick clip — SliceKernel records slicers, input_grid, and the output grid. Given the output brick coordinate, clip derives element bounds from both grids’ slice_ranges, translates output positions through the slicer (start + pos·step) into source space, subtracts the input brick’s element start to get brick-local ranges, and slice_each_axises the dense payload. A Fill input short-circuits: re-emit a Fill payload at the output brick shape — no element work.

Coarsen

There is no coarsen op. a.coarsen({"i": 2}).sum() builds ArrayOp::Reduce { acc, view: {"i": Frame::Window { size: 2, stride: 2 }} } — a framed reduce, where array::expr::Frame is in element space. Lowering emits Op::Reduce(LoweredAccumulator):

  • Output chunk grid — ChunkGrid::after_reduce maps each viewed axis: Frame::WholeAxis { keepdims: false } drops it, keepdims: true keeps it at size 1, and Frame::Window with size == stride coarsens the axis chunks (AxisChunks::coarsen, extent ceil(n / size)).
  • Edge encoding — build_reduce_access re-derives the element-space frame in chunk space. With w = ceil(n_in_chunks / n_out_chunks):
    • w == 1: the window fits inside one brick — identity edge (no map entry). The frame is invisible to the planner; all folding is element-space inside the accumulator.
    • w > 1: the window spans bricks — AxisAccess::Composed(brix::Frame::Window { size: w, stride: w }). This is the Composed arm: not a runtime coord override but loop structure. The planner’s extend_iter_space_for_fold logical_divides the axis into (n_out, w) and appends a fold loop over the inner mode; a non-divisible n_in records a predication limit so the trailing partial window doesn’t read past the producer extent.
  • Within-brick fold — LoweredAccumulator carries the element-space view and a keepdims=true state grid (the reduced axis survives at size 1 in intermediate state regardless of the user’s keepdims). fold_left re-frames each incoming brick (as_frame_iter) and calls the inner accumulator’s fold_frame per (output lane, frame slot); finalize squeezes out axes whose view was WholeAxis { keepdims: false }.

The two regimes are why the visualizer’s coarsen workload uses one-element bricks: it forces w = 2 so the Win(2,2) edge and the planner-generated fold loop become visible. With bricks at least as large as the window the same expression lowers to an identity edge.

Vindex

ArrayOp::Vindex(VectorizedIndexer) — equal-length integer index vectors per consumed axis, zipped into a single produced axis k — lowers to Op::Blockwise(VIndexKernel).

  • Run partition — plan time, in chunk_grid::vindex::VectorizedSelection::from_indexer. Walk the k positions; AxisChunks::locate resolves each index into (source_chunk, in_chunk_position) per consumed axis. Consecutive positions sharing the same multi-axis chunk-coord tuple group into a run; the tuple sequence must be non-decreasing in lex order over the (sorted) consumed axes — hard error otherwise, no auto-sort. The result is VectorizedSelection { consumed_axes, per_axis } where per_axis[a][r] is an AxisSelection { source_chunk, positions } for axis a, run r.
  • Output chunk grid — output_grid_and_selection removes the consumed axes (without_axes) and appends k as AxisChunks::Variable(run_widths): one output chunk per run, so each output brick reads exactly one input brick.
  • Edge encoding — keyed by the produced axis, which does not exist on the producer grid: {"k" → AxisAccess::Translated(TranslatedAccess::Zipped({i: AxisOverride::Lookup(tᵢ), j: AxisOverride::Lookup(tⱼ)}))}. Each follower’s table maps run index → that axis’s source chunk. At runtime resolve_coord reads the leader k’s iter coord from the enclosing loop and drives every follower’s Lookup from it — the co-iteration that distinguishes vindex from oindex, whose per-axis TranslatedAccess::Independent(AxisOverride::Lookup) entries form a cartesian product.
  • Within-brick gather — VIndexKernel records the VectorizedSelection plus both grids. For output run r, it loops i over the run width, index_axis_moves the source brick at per_axis[a][r].positions[i] on every consumed axis simultaneously, and assigns the resulting cell into lane i of the produced axis. Fill inputs short-circuit to a Fill at the output brick shape.

Implementation tour

1. Foundations

Read first — every other module uses these.

OrderModuleWhy
1src/brix/error.rsOne error enum, used everywhere. Quick read.
2src/brix/brick.rsBrick, BrickCoordinate, BrickPayload. The unit of data that flows through every layer.
3src/array/brick.rsDense and Fill payload impls. Concrete f64 payloads today; will become generic when integer/bool dtypes land.
4src/brix/iteration_space.rs and src/array/chunk_grid/mod.rsTwo flavours of chunk topology. brix::IterationSpace is counts-only (orchestration view); array::ChunkGrid is the rich per-axis representation (AxisChunks::Uniform / Variable / future Sharded).

2. The Brix-IR contracts

Read second — these are the trait surfaces the rest of the system targets.

OrderModuleWhy
5src/brix/expr.rsThe Source / Kernel / Accumulator traits, the Op enum, the Expr node. This is the structural vocabulary the planner consumes.
6src/brix/sink.rsThe Sink trait. Concurrency-safe (&self on write_brick); not part of the DAG.
7src/brix/cost.rsCost struct and how nodes report it. The planner aggregates this; understanding the shape clarifies what each Source::cost / Accumulator::cost impl is computing.

3. Concrete impls

Now look at the built-in Source / Kernel / Accumulator impls. Each is self-contained and small.

OrderModuleWhy
8src/array/source.rsFillSource, DenseSource, LoweredOpaque. Note the OpaqueSource extension trait — this is the user-facing API for Zarr / Arrow IPC integrations.
9src/array/kernels/ops.rsUnaryKernel, BinaryKernel. The simplest kernels — pure elementwise.
10src/array/kernels/indexing.rsSliceKernel, GatherKernel. More involved; introduces the coord-rewriting story.
11src/array/accumulators.rsSumAccumulator, MeanAccumulator. Read for the State downcasting dance (downcast_state_mut / downcast_state_ref) — that’s the cost of Arc<dyn Accumulator> heterogeneity.

4. The array surface

OrderModuleWhy
12src/array/expr.rsArrayExpr and ArrayOp. The logical IR before chunking decisions. Carries ArrayLayout (named axes + extents + strides).
13src/array/ops.rsUnaryOp / BinaryOp enums and their evaluation. Element-level semantics, distinct from Kernel impls.
14src/array/indexer.rsIndexer — how slicing and gather get expressed at the array level before lowering.
15src/array/mod.rsThe user-facing Array handle. Skim the impl block to see what operations exist; the tests at the bottom show end-to-end flow.

5. The lowering pass

OrderModuleWhy
16src/array/optimize.rsPre-lowering rewrites on ArrayExpr (e.g., adjacent-select fusion). Small.
17src/array/lower.rsThe interesting one. Walks ArrayExpr post-order, picks ChunkGrid per node, projects to brix::IterationSpace, and assembles Arc<Expr>. All chunk-sizing logic lives here.

6. Planning and execution

OrderModuleWhy
18src/brix/expr_graph.rsBuilds a graph view (Node + Edge) of an Arc<Expr> DAG. The planner walks this rather than the Arc-based tree.
19src/brix/planner.rsTurns the expr graph into a PhysicalPlan. Reads AccessRelations to decide loop order and stage boundaries.
20src/brix/schedule.rsThe Schedule tree — Loop, Sequence, Eval, Init, Finalize, WriteSink. The schedule! macro is here. Tree is data, not code.
21src/executor/sync.rsInterpreter for Schedule. A few hundred lines. Read last — by now every term in it should be familiar.

7. Sinks and integration

OrderModuleWhy
22src/array/sink.rsInMemorySink. Read with Sink trait open in another tab. Demonstrates the &self write contract — the executor can call it from multiple threads.
23tests/exercises.rs and tests/common/mod.rsEnd-to-end exercises that thread every layer together. Run them with RUST_LOG=trace cargo test to see the planner and executor narrate.

What to skip on first pass

ModuleWhy skip
src/brix/layout.rsCuTe-style layout algebra. Large; only the surface (Layout, NamedLayout, native_layout) is needed for first-pass understanding.
src/array/chunk_grid/{oindex,select}.rsTwo specific chunk-grid rewrites. Read after lower.rs if you’re touching indexing.
Anything in checkpoints/Design rationale and decision history. Use as reference when the why of a design choice isn’t clear; not load-bearing for understanding the what.

Suggested entry points by task

  • “I want to add a new operation” → read 5, 9, 13. Add a Kernel impl in array/kernels/, plumb through ArrayOp and lower.rs.
  • “I want to add a new dtype” → read 2, 3, the DType enum in array/dtype.rs. Touchpoints: every Dense / Fill site, to_dense_f64 (will become dtype-aware), payload to_bytes / from_bytes.
  • “I want to add a new accumulator” → read 5, 11. Copy SumAccumulator, change the fold logic, add an output_dtype impl, register in lower.rs.
  • “I want to integrate a new storage backend” → read 6, 8 — implement OpaqueSource in array/source.rs. The lowering machinery wraps it in LoweredOpaque automatically.
  • “I want to add an optimization rewrite” → read 16, 17. Decide whether your rewrite is structural (greedy pass, like fuse_adjacent_selects) or algebraic (future egglog). For structural, copy the pattern in optimize.rs.
