The Brix Book

Data Types

Requirements

  • We need to support something like Datetime { calendar, epoch, units }. We expect an external zax-cftime crate to provide implementations for non-standard calendars (since this is a climate science-only requirement).
  • We need to support Strings.
  • We need to support Arrow geometry buffers. We can ignore arithmetic ops here but need to support slicing.

Approach

Warning

The dtype model is in flux as we work toward Zax (zax-io) integration. The shipped code is still the flat DType { F64, I64 } enum; the shape below is the migration target. Everything past the primitives is design intent — see checkpoints/dtypes.md.

Brix is statically typed: every array carries a DType, and scalars are tagged Values (src/array/dtype.rs).

  • Physical dtype — dtype of the runtime brick payload. These are primitive types (i64, f64 etc.).
  • Logical dtype — what the values mean (DType): WE can wrap primitive types Primitive, and defined derived types on top of primitive types: Datetime (over i64), Categorical, or other types: Utf8, Extension.
  • Validity — a null mask parallel to the value buffer (discussed in Payloads). Dtypes don’t carry “nullability” (like Polars).

Logical DTypes

A Primitive(PrimitiveType) variant wraps the closed primitive set, core-recognized logical types are their own variants, and Extension is the opaque escape hatch.

pub enum PrimitiveType {
    F64,
    I64,
}

pub enum DType {
    Primitive(PrimitiveType),              // F64/I64 (F32, U8 … later)
    Datetime(Arc<dyn DatetimeLikeDtype>),  // unified numpy + CF time; physical = I64
    Utf8,                                  // variable-length strings; String kind
    Extension(Arc<dyn ExtensionDType>),    // opaque (geometry); physical = None
}

Datetime

We mode datetimes as one logical dtype backed by one trait. All datetimes are characterized by calendar, unit, and epoch. E.g. numpy’s datetime64 has fixed calendar = proleptic_gregorian and a unix epoch. “cftime” provides non-standard calendars and an arbitrary epoch. We will represent all these as i64 under the hood. We expect to have an external zax-time (or at least zax-cftime) so the core library defines traits (DatetimeLikeDtype) and enums (TimeUnit)

DType::Datetime(Arc<dyn DatetimeLikeDtype>)

pub trait DatetimeLikeDtype: Send + Sync + Debug {
    fn unit(&self) -> TimeUnit;                       // S/Ms/Us/Ns/D…
    fn storage(&self) -> PrimitiveType;               // = I64; feeds physical()
    // calendar/tz-dependent behavior — the reason this is a trait:
    // time-specific ops, e.g. for resampling
}
  • Datetime arithmetic is implemented on the i64 kernel. dt − dt, dt + td, dt < dt are calendar-independent by definition; only the output dtype tag changes. Cross-unit (dt[s] + td[ns]) is a plan-time unit-scaling AsType.
  • Calendar-dependent ops (add months/years, resample buckets, decode, components) are deferred to specific external implementations (this needs to be sketched out later)
  • Timezone is metadata on the gregorian calendar impl.

String

The logical dtype is Utf8; its physical layout (offsets + bytes) lives in a separate brick kind — see the String kind in Payloads.

Value

A typed scalar — used for fill values, reduction identities/absorbers, and constants:

/// Typed scalar.
#[derive(Debug, Clone, Copy, PartialEq)]
pub enum Value {
    F64(f64),
    I64(i64),
}

Promotion

DType::promote defines a NumPy-style symmetric promotion rules. Per-op output dtypes (e.g. mean → f64) live on the ops themselves, not here. A mixed-dtype binary op (e.g. int + float) inserts a plan-time AsType node (see The Array IR). This allows a “relatively simple” implementation of unary ops

Typed buffers

DType/Value describe a single element; a brick’s worth of dense values is held in a TypedArray — a dtype-tagged ArrayD<T>. It’s the dense container only; sparse / extension kinds carry their own representation.

/// Dense-kind value buffer, tagged by dtype. This is the dense container only —
/// `ArrayD<T>` per primitive — not a universal typed buffer (sparse / extension
/// kinds carry their own representation; see `checkpoints/dtypes.md`).
#[derive(Debug, Clone)]
pub enum TypedArray {
    F64(ArrayD<f64>),
    I64(ArrayD<i64>),
}

Borrowed access goes through TypedView, the view-mirror of TypedArray (the Dense arm of a ReadFrame). A consumer that knows its concrete T recovers an ArrayViewD<T> via TypedView::as_typed.

/// Borrowed, dtype-tagged view — the `Dense` arm of [`ReadFrame`]. Mirrors
/// [`TypedArray`] for views. Consumers that know their `T` recover a concrete
/// `ArrayViewD<T>` via [`TypedView::as_typed`].
#[derive(Debug, Clone)]
pub enum TypedView<'a> {
    F64(ArrayViewD<'a, f64>),
    I64(ArrayViewD<'a, i64>),
}

Parameterizing over dtypes

DType/Value/TypedArray are runtime-tagged — a closed enum per concept. Generic code, however, wants to be written once over a static type parameter T. The Element trait is the bridge: it is implemented per primitive, and its const DTYPE ties the static T to the runtime tag.

pub trait Element: Copy + Send + Sync + Debug + PartialEq + 'static {
    const DTYPE: DType;
    fn from_array(arr: ArrayD<Self>) -> TypedArray;
    fn from_scalar(v: Self) -> Value;
    fn from_value(v: Value) -> Option<Self>;
    fn to_le_bytes(self) -> [u8; 8];
    fn from_le_bytes(b: [u8; 8]) -> Self;

    // -- type-erasure bridge --------------------------------------------------
    // These let `dispatch_dtype!`/generic bodies recover a concretely-typed
    // view/buffer out of the runtime-tagged `TypedView`/`TypedArray` enums.
    // A generic fn can't coerce a `TypedView::F64` arm into `ArrayViewD<T>`,
    // but a *concrete* impl (T = f64) can — so the peel lives here.

    /// Wrap a concrete view back into the runtime-tagged `TypedView`.
    fn wrap_view(v: ArrayViewD<'_, Self>) -> TypedView<'_>;
    /// Peel a `TypedView` to this element's view, if the tag matches.
    fn view_from_typed(tv: TypedView<'_>) -> Option<ArrayViewD<'_, Self>>;
    /// Mutable borrow of the inner buffer, if the tag matches.
    fn array_mut_from_typed(ta: &mut TypedArray) -> Option<&mut ArrayD<Self>>;
    /// Take the inner buffer by value, if the tag matches.
    fn array_from_typed(ta: TypedArray) -> Option<ArrayD<Self>>;
}

The trait carries a type-erasure bridge: a generic fn can’t coerce a TypedView::F64 arm into ArrayViewD<T>, but a concrete impl (T = f64) can — so wrap_view / view_from_typed / array_from_typed live on Element, where T is known. Runtime dispatch in the other direction (tag → concrete body) goes through the with_typed_array! / dispatch_dtype! macros in dtype.rs.

Capability traits

Op kernels don’t bound on Element directly — they bound on capability traits that bundle exactly the arithmetic a body needs. Binary (add / sub / mul plus the Zero/One reduction identities) is unsigned-compatible; Unary (negation, abs) is signed-only — Signed supplies both, so a future u8 is correctly excluded. Numeric is the both-families bundle, granted free by a blanket impl:

/// Capability umbrella for the `BinaryOp` family (add / sub / mul) plus the
/// `Zero`/`One` identities reductions need. **Unsigned-compatible** — a future
/// `u8`/`u16` code-width dtype implements this, so binary ops / `sum` / `cast`
/// stay open to it. The universal arithmetic bound; mirrors `BinaryOp`.
pub trait Binary:
    Element + Add<Output = Self> + Sub<Output = Self> + Mul<Output = Self> + Zero + One
{
}

/// Capability for the `UnaryOp` family (negation / absolute value). **Signed
/// only** — `Signed` supplies both `Neg` and `abs`, so unsigned dtypes are
/// correctly excluded. Mirrors `UnaryOp`.
pub trait Unary: Element + Neg<Output = Self> + Signed {}

/// Fully numeric: both op families. The convenience bundle for code that needs
/// everything; bodies that need only one family should bound on [`Binary`] or
/// [`Unary`] directly so they don't over-constrain (e.g. exclude unsigned).
pub trait Numeric: Binary + Unary {}

// Concrete `impl Element/Binary/Unary for {f64,i64}` blocks live in
// `array/generated/dtypes.rs` — one mechanical block per primitive dtype.

// `Numeric` is a method-less bundle, so a blanket impl gives it for free to any
// type that has both capabilities — no per-dtype `impl Numeric` line needed.
impl<T: Binary + Unary> Numeric for T {}

All three are method-less: the behavior comes entirely from the supertrait bounds, so the per-primitive impls in generated/dtypes.rs are empty markers (impl Unary for f64 {}). A unary op runs one monomorphic body bounded <T: Unary>, reached by dispatch_dtype! on the runtime tag — Neg => -x, Abs => x.abs().

Where per-dtype code lives

The contracts — Element (and Unary/Binary), the DType/Value enums, and the dispatch macros — live in array/dtype.rs. The concrete impl Element for f64 { … } / for i64 { … } blocks are mechanically instantiated in array/generated/dtypes.rs — one block per primitive, the file you extend when a new dtype lands. (Folder convention, not codegen.)

Extensions

  • Strings
  • Arrow geometry buffers
  • CFTime
  • Interval Arrays
  • Categorical Arrays
  • Sparse Arrays

Two of these are worked through in Extensions; their dtype treatment differs:

Geometry is an extension dtype: DType grows an Extension(Arc<dyn ExtensionDType>) arm whose physical() returns None — opaque, so it never enters dispatch_dtype!/with_typed_array!; geometry kernels live in zax-geo and downcast the payload. Type identity (extension name + CRS) drives the Eq/Hash alignment check, the same role DType::F64 plays for native arrays.

Sparse is not a dtype — it’s a physical kind (a BrickPayload sibling of Dense/Fill). A sparse brick still carries an ordinary values dtype (physical() = Some(F64) etc.), so its nonzero values ride the normal Element/promotion path: a sparse-i64 × dense-f64 op gets the usual plan-time AsType before the kernel runs. Only the structure (nonzero pattern, formats) lives behind dyn SparseArray.
