The Brix Book

Goals

Brix is an nD chunked-array computing system built in Rust. It is intended to power analytic workflows classically driven by Xarray/Dask, bringing in ideas from OLAP databases, and also advance the state-of-the-art by innovating on interfaces and data models.

The problems

A few problems with the current Python data analysis stack from years of experience:

  • Close co-operation between compute and storage layers can unlock a lot.
    • DB systems deliver a lot of value through such co-operation.
    • This includes “materialized views”, “incremental view maintenance”, and storing pre-computed statistics of the data to accelerate downstream queries. None of these valuable capabilities are meaningfully available to the scientific Python array computing community.
  • We leave a lot of performance on the table today.
    • Optimizing workloads is extremely hard. Because the code is divided between many independent OSS projects, a lot of context ends up being lost when communicating between layers. Good performance requires heroic engineering in multiple projects, some of which are more active than others.
    • The optimization point is extremely important, now that we have Icechunk which has the ability to saturate network hardware. We need a compute stack that can take full advantage of this.
  • Users are not isolated from the storage layer and are forced to consider “chunk sizes” at open-time. This is a major drag on performance, and user-friendliness.
  • Many datasets, particularly in Earth Sciences, are streaming (albeit relatively slow). We want to support incremental, streaming compute in the future. Nothing in the Python stack today was designed with this in mind.
  • At its lowest level, the current python stack is a wrapper over C-level loops. This makes extensibility quite hard. The Numpy ufunc API is great e.g. (cumulative max =np.maximum.accumulate, grouped max = np.maximum.at) but fails at composing these primitives, requiring arcane workarounds (how do you run an cumulative max aggregation within groups?). Doing such complex operations usually involves using a compiled extension point (Numba/Rust/Pybind11 etc.). Such composition is required for many climate analytics workloads.

Design goals

  1. Predictable execution: either execute or fail hard if query cannot be satisfied with defined (mem/cpu/time etc.).
  2. Isolate the user from chunk-size choices through cost-based planning.
  3. Easy execution next to the data: Separate execution from expression, allowing remote execution of the workload and return results. Following Halide, what to compute (the Expr graph) should be separated from how (the Schedule).
  4. Support input data from multiple data formats:
    1. At least Icechunk for array inputs
    2. Some tabular format for tabular inputs, including geometries for zonal statistics workloads.
  5. All interfaces should be designed so that AI can generate readable/skimmable/auditable code i.e. provide high-level primitives.
    1. Xarray-like: Rich interface that uses metadata and exposes n-dimensionality of the underlying data.
    2. SQL: must be able to accept queries expressed in SQL and return results in a tabular form
  6. Be as familiar as possible. To ease adoption, we will strive to provide an API surface familiar to existing Python data stack users, but will absolutely innovate on API.
  7. Usable from multiple languages : Python + Rust
  8. Support UDFs in both Rust and Python: key to supporting arbitrary user workloads
  9. Strong support for data schemas
  10. Export results to zero-copy standard interchange formats: DLPack + Arrow
    1. This is required so we can offer to_xarray or to_polars as an escape hatch
  11. Be able to saturate the machine
  12. Design around static shapes with explicit escape hatches. To start with, we assume that all extents are known at plan time. Data-dependent shapes (dropna, unique, boolean indexing) go through explicit materialization barriers. In practice, since Xarray requires static shapes, this isn’t a strong limiter on workloads.
  13. Explicit over implicit. Hard errors and limits over heuristics; user-controlled materialization hints over automatic CSE.
  14. Be amenable to developing incremental compute features in the future.
  15. Query graphs should be serializable to enable stored views.

Inspirations

The tabular data world has undergone a major transformation, with projects like Apache Arrow, Polars, DuckDB, Ibis, and many others innovating around performance and scalability of analytics tools. Apache DataFusion in particular has rolled up decades of accumulated knowledge about analytical query execution into a developer-friendly package. Overall these trends have favored:

  1. Compiled languages (Rust, C++) with multi-language bindings (Python, R, etc.)
  2. Performant multi-threaded single-node execution over complex distributed clusters
  3. SQL as a lingua franca for analytics

An equally vibrant array compiler / DSL space has also spun up in the last decade targeted at ML teams, not SaaS. MLIR/XLA/Triton/CuTile/”Tensor Comprehensions”/Halide/Mojo/Finch.jl/TACO all have the very similar goals - write once, execute optimally on different hardware. These projects are unifying 50 years of HPC work on matrix computation (e.g. BLAS) with more generic array transformations needed for ML kernels.

Xarray, Dask, and the Pangeo stack have not kept pace, with limited major architectural innovations over the past decade.

Architecture

Brix is a layered nD chunked-array compute orchestration system.

Concepts

At a high level there are three layers:

  • the array layer characterized by shape, dtype, chunk grid with chunk sizes, and dimension names
  • the “brix” layer but with arrays abstracted away) characterized purely by a “brick grid” with a shape but no per-brick size.
  • the “schedule” layer that translates a brix-level DAG to an iterator-level description consumed by an executor.

Think of the “array layer” as expression “what to compute” in array-element space (e.g. rolling window over 5 elements along axis 'x').

Think of the “brix layer” as representing a wiring diagram for routing chunks bricks; to do so we do not care about the contents of each brick, just that we know where to route it. This is again “how to compute” but with actual array contents abstracted away.

Think of the “schedule layer” as expressing “how to compute” with limited degrees of freedom exposed to the executor.

Note

We are using the term “brick” intentionally to avoid confusion with “chunks” and “shards” as used in the storage layer (e.g. Zarr). “Tile” is another term used commonly.

The brick grid is for orchestration, it need not be identical to or even aligned with the “chunk” or “shard” grid of the dataset (though of course there are benefits to doing so).

Each downward step is a single-purpose pass; each layer carries strictly less semantic information than the one above it.

The high-level flow is as follows:

  • users create Array objects (either from a Source) or an in-memory ndarray.
  • operations are expressed using familiar methods on Array objects e.g. Array::over(axes).sum().
  • under the hood, these operations are recorded using a lower-level ArrayExpr with a much smaller API surface. For example, Array::over(axes).sum() is really an ArrayOp::Reduce { acc, view }, carrying the accumulator and the per-axis frame (view).
  • At this point there are no “chunk sizes” or “brick grids” associated with Array.
  • When ready to compute, we first run an optimization pass on the ArrayExpr (e.g. pushing indexing operations as low as possible).

Choices

Three-layer separation

Choosing the array+brix+iteration layer is a choice.

Separating the Array from Brix (Iteration Space/chunk grid operations) has some benefits. It makes it clear that the core orchestration is manipulating arbitrary data payloads over a iteration space (‘brix grid’) that only cares about the Cost per brick rather than details (dtype/chunk size/ etc.) This leaves us open to orchestrate other arbitrary data payloads in the future, for example orchestrating a grid of filenames for ingest. As prior art, Xarray uses dask as a pure task scheduling engine precisely at ingest time, where manipulating a list of filenames in parallel is necessary to construct a data cube.

An implication that has both down-side and up-side is that the same concept is commonly expressed at both the Array layer and the Brix layer (e.g. sliding windows/slicing). This can get confusing. The up-side though is that it’s easy to spot when we (really agents) mix array-layer concerns in to the brix-layer. Personally, I have found it far easier to reason about the code once I added the separation.

Choosing to have an iteration layer / Schedule is a choice that comes from realizing that in most array workloads we have a lot of information available at plan-time (see below).

Execution Model

We are choosing to have the Executor be an interpreter of the Schedule.

DB query engines express pipelines through a runtime protocol between operators: pull (Volcano next()) or push (consume()). Both approaches converge to nearly the same thing. (Shaikhha et al. 2016).

In the case of arrays, shapes, chunk grids, and iteration spaces are known at plan-time. So the planner emits the fused loop nest directly. Co-iteration of multiple inputs — which forces pull engines to coordinate demand and push engines to buffer one side — is resolved at plan-time and recorded using access relations along the edge. This kind of co-iteration is important for broadcasting e.g. A[i,j] + B[j]. At plan-time, we can hoist B’s read above the i loop and reuse the brick across iterations.

DB engines achieve the same reuse, but indirectly: no operator knows which loop level it sits at, so plans insert materialization instead (Postgres Materialize/Memoize, a hash join’s build side) — caching B wholesale and paying lookup cost, rather than holding one brick at the right loop depth.

This kind of “schedule” is common to array compilers, most clearly Halide, whose algorithm/schedule split TVM and TACO inherit. Tensor Comprehensions makes the same separation in polyhedral form — its schedules are the ISL schedule trees cited in Planner. Chapel’s domain maps are kindred: they factor data distribution and iteration out of the loop body.

By contrast, Dask bakes iteration into the materialized task graph (one task per chunk — iteration is the graph). Cubed keeps its plan at the op level and generates per-chunk tasks lazily, but hardcodes the schedule — an unordered, embarrassingly-parallel map over output blocks — rather than representing one that can be manipulated.

Arrays

Concept

All arrays are characterized by:

  1. dtype - A DType
  2. shape - a tuple of ints
  3. dims - Dimension names

An Axis is characterized by name and extent.

Important

Arrays have no chunk sizes (or chunk grids) associated with them!

Arrays are a thin wrapper on top of ArrayExpr presenting a more familiar, broader API surface to the user. At lowering time, an explicit ChunkGrid is chosen for each Array node in the graph based on cost considerations. At compute-time, arrays are conceptualized as streams of ArrayPayload bricks that are operated on. These ArrayPayload bricks are wrappers around underlying bare arrays (e.g. ndarray::ArrayD).

Kinds

We include two concrete arrays motivated by Zarr’s storage model where fill_value chunks need not be stored on disk. Both kinds satisfy the ArrayLike trait — one required method, expr(), with the rest defaulting to read through that expression (definition pulled live from src/array/arrays/mod.rs):

/// Trait for user-facing array variants ([`Array`], [`BlockSparseArray`]).
/// Required method: [`Self::expr`].
/// Everything else has a default that reads through the expr.
pub trait ArrayLike: std::fmt::Debug + Send + Sync {
    /// The array-level IR node this array wraps.
    fn expr(&self) -> &Arc<ArrayExpr>;

    fn shape(&self) -> Vec<usize> {
        self.expr().shape()
    }

    fn dtype(&self) -> DType {
        self.expr().dtype
    }

    fn dims(&self) -> Vec<&str> {
        self.expr().dims().iter().map(|s| s.as_str()).collect()
    }

    /// Run array-level optimization passes.
    fn optimize(&self) -> ArrayResult<Array> {
        Ok(Array::from(optimize::optimize(self.expr())?))
    }
}

We decompose to two axes:

  1. a dense or sparse chunk grid (“iteration space”);
  2. dense / sparse array (whether fill_value is a semantically valid value that requires propagation).

For example, consider adding 1 to an array backed by a Zarr OpaqueSource where some on-disk chunks are missing / interpreted as all fill-value. If the fill_value is semantically invalid (e.g. NaN) then:

  1. we needn’t do anything at all for fill-value chunks.
  2. we add 1 to every populated chunk. If the fill_value is semantically valid (e.g. 0) then
  3. we simply need to update the fill_value to 0+1=1.
  4. we add 1 to every populated chunk.

We model:

  1. the underlying chunk grid as dense/sparse (translates to a dense/sparse iteration space at the brix layer) using brix::Occupancy in both ChunkGrid and brix::IterationSpace.
  2. model the presence of a valid fill_value through a BlockSparseArray that adds a new fill value property to Array.

Choice

BlockSparseArray requires explicit fill_value treatment and propagation much in exactly the same way that sparse matrices do. Thus it makes sense to split that up from arrays where the fill_value is invalid and no such logic is required so I chose to not use fill_value: Option<Value> on Array.

The Array IR

Array is a thin handle — the operations you call on it are recorded into ArrayExpr: the array-layer IR (src/array/expr.rs).

An ArrayExpr node is its op plus its inputs and the metadata that makes the layer labelled and dtype-aware.

/// Array-level expression node.
#[derive(Debug, Clone)]
pub struct ArrayExpr {
    pub op: ArrayOp,
    pub inputs: Vec<Arc<ArrayExpr>>,
    // array shape, strides, order
    pub layout: ArrayLayout,
    pub dtype: DType,
    // currently unused, but anticipating usefulness of distinguishing
    // sparse/dense arrays for planning/optimizer purposes
    pub brick_kind: ArrayPayloadKind,
    pub label: Option<String>,
}

Each node wraps an ArrayOp. The op set is deliberately explicit — separate Unary/Binary rather than a single general Map — while the optimizer’s needs are still being worked out:

/// Array-level op.
/// These are pretty explicit at the moment anticipating that matching
/// against specific variants are useful for optimization.
/// i.e. Unary, Binary are separated out, and not just a general Map.
#[derive(Clone)]
pub enum ArrayOp {
    // Transparent sources are _not_ chunked and any chunk grid may be imposed.
    // E.g. in-memory arrays
    TransparentSource(TransparentSource),
    // Opaque sources are chunked and have a chunk preference
    // E.g. Zarr/Icechunk
    OpaqueSource(Arc<dyn OpaqueSource>),
    Unary(UnaryOp),
    Binary(BinaryOp),
    Reduce {
        acc: Arc<dyn ArrayAccumulator>,
        /// Frame applied to each reduce axis
        view: HashMap<AxisName, Frame>,
    },
    /// Per-axis positional slicing. One `Slice` per indexed axis.
    Slice(HashMap<AxisName, Slice>),
    /// Per-axis orthogonal integer indexing.
    Oindex(HashMap<AxisName, Vec<usize>>),
    /// Lined-axis vectorized integer indexing.
    Vindex(VectorizedIndexer),
    /// Cast to a target dtype. Plan-inserted for implicit NumPy-style
    /// promotion (see [`ArrayExpr::binary`]) or user-driven via `Array::cast`.
    Cast(DType),
    // Future: Transpose, Reshape, ...
}

Promotion

DType::promote defines a NumPy-style symmetric promotion rules. Per-op output dtypes (e.g. mean → f64) live on the ops themselves, not here. A mixed-dtype binary op (e.g. int + float) inserts a plan-time Cast node (see The Array IR). This allows a “relatively simple” implementation of unary ops

Typed buffers

DType/Value describe a single element; a brick’s worth of dense values is held in a TypedArray — a dtype-tagged ArrayD<T>. It’s the dense container only; sparse / extension kinds carry their own representation.

/// Dense-kind value buffer, tagged by dtype. This is the dense container only —

#[derive(Debug, Clone)]
pub enum TypedArray {
    F64(ArrayD<f64>),
    I64(ArrayD<i64>),
}

/// Borrowed, dtype-tagged view — the `Dense` arm of [`ReadFrame`]. Mirrors

pub enum TypedView<'a> {

Parameterizing over dtypes

DType/Value/TypedArray are runtime-tagged — a closed enum per concept. Generic code, however, wants to be written once over a static type parameter T. The Element trait is the bridge: it is implemented per primitive, and its const DTYPE ties the static T to the runtime tag.

pub trait Element: Copy + Send + Sync + Debug + PartialEq + 'static {
    const DTYPE: DType;
    fn from_array(arr: ArrayD<Self>) -> TypedArray;
    fn from_scalar(v: Self) -> Value;
    fn from_value(v: Value) -> Option<Self>;
    fn to_le_bytes(self) -> [u8; 8];
    fn from_le_bytes(b: [u8; 8]) -> Self;

    // -- type-erasure bridge --------------------------------------------------
    // These let `dispatch_dtype!`/generic bodies recover a concretely-typed
    // view/buffer out of the runtime-tagged `TypedView`/`TypedArray` enums.
    // A generic fn can't coerce a `TypedView::F64` arm into `ArrayViewD<T>`,
    // but a *concrete* impl (T = f64) can — so the peel lives here.

    /// Wrap a concrete view back into the runtime-tagged `TypedView`.
    fn wrap_view(v: ArrayViewD<'_, Self>) -> TypedView<'_>;
    /// Peel a `TypedView` to this element's view, if the tag matches.
    fn view_from_typed(tv: TypedView<'_>) -> Option<ArrayViewD<'_, Self>>;
    /// Mutable borrow of the inner buffer, if the tag matches.
    fn array_mut_from_typed(ta: &mut TypedArray) -> Option<&mut ArrayD<Self>>;
    /// Take the inner buffer by value, if the tag matches.
    fn array_from_typed(ta: TypedArray) -> Option<ArrayD<Self>>;
}

The trait carries a type-erasure bridge: a generic fn can’t coerce a TypedView::F64 arm into ArrayViewD<T>, but a concrete impl (T = f64) can — so wrap_view / view_from_typed / array_from_typed live on Element, where T is known. Runtime dispatch in the other direction (tag → concrete body) goes through the with_typed_array! / dispatch_dtype! macros in dtype.rs.

Sources & Sinks

Sources

The array layer has two flavors of source, split on whether the data has an inherent chunking.

A transparent source has no inherent chunk grid — lowering may impose any brick shape on demand:

/// Transparent sources — optimizer can pattern-match on the variant.
/// No inherent chunk grid; the planner (via lowering) assigns one freely.
#[derive(Debug, Clone)]
pub enum TransparentSource {
    Fill { value: Value, shape: Vec<usize> },
    Dense(TypedArray),
    // Future: Repeat { repeats: usize, axis: usize }, Tile { ... }
    // and Functional { Fn ... }
}

An opaque source (Zarr-like) owns its native chunk grid which may carry occupancy for block-sparse data:

/// Opaque source — I/O-backed (Zarr, Arrow, etc.).
/// The optimizer can't see through this; it only knows shape, layout, and
/// `ArrayPayloadKind`. The source owns its chunk grid (data is already chunked
/// on disk).
///
/// Lowering wraps an `Arc<dyn OpaqueSource>` in a [`LoweredOpaque`]
/// before handing it to the planner.
pub trait OpaqueSource: Debug + Send + Sync {
    fn shape(&self) -> &[usize];
    /// The source's chunk grid — opaque sources own theirs (already
    /// chunked on disk / in storage). Lowering reads it to project a
    /// `brix::IterationSpace` and bake it into the [`LoweredOpaque`] wrapper.
    fn native_grid(&self) -> &SourceNativeChunkGrid;
    // Grab data for this range in array index space.
    fn read(&self, bounds: &[Range<usize>]) -> BrixResult<Arc<dyn BrickPayload>>;
    fn cost(&self, coord: &BrickCoordinate) -> Cost;

    /// Element dtype the source produces.
    fn dtype(&self) -> DType;

    fn brick_kind(&self) -> ArrayPayloadKind;
}

Lowers to a FillSource / DenseSource, LoweredOpaque, all of which include the chosen ChunkGrid and impl the brix::Source trait. Note that the chosen ChunkGrid need not align with the OpaqueSource’s native chunk grid.

Aggregations

Aggregations are (at the moment) reductions over one or more axes — sum, mean, and friends. They are backed by accumulators and computed over a frame.

extern crate brix;
use std::collections::HashMap;
use brix::Array;

fn main() -> Result<(), brix::array::ArrayError> {
let data = Array::fill(1.0, vec![3, 4], &["time", "x"]);
// Reduce over the whole `time` axis.
let totals = data.clone().over(&["time"])?.sum()?;
// Coarsen `x` into width-2 blocks, then average each block.
let coarsened = data.coarsen(HashMap::from([("x", 2)]))?.mean()?;
Ok(())
}

Frames

A frame is a lens over an array: a way of viewing it as a stream of windows rather than as a flat block (src/array/framed.rs). The framing model borrows from two places:

  1. numpy ufunc methods — reduce collapses a whole axis (WholeAxis), reduceat reduces over segment boundaries (Blocks/VariableWindow), and at scatters into indexed groups (Groups). A Frame is the descriptor that picks which behavior the same accumulator runs. Unlike ufuncs, our intent is to allow scans (“cumulative ops”, accumulate) to compose with these Frames.
  2. SQL:2003 window functions — an aggregate evaluated against a moving frame rather than the whole input, spelled as an OVER clause (SUM(x) OVER (...)) and optionally factored out as a named WINDOW W AS (...). Brix’s over plays the role of OVER, and a single frame model subsumes both halves of the clause: PARTITION BY grouping is Groups (per-element labels), while the frame spec is WholeAxis, Window, or Blocks.

All of these are just frame shapes fed to the same accumulator machinery.

/// This enum enables our approach to composing Aggregators (Reduce, Scan) with
/// different _views_ or _frames_ along an axis
/// (`rolling`, `groupby`, `coarsen`, etc.)
#[derive(Clone, Debug, PartialEq, Eq)]
pub enum Frame {
    /// Reduce the whole axis; preserve size-1 axis if keepdims is true
    WholeAxis { keepdims: bool },

    /// Regular contiguous frames.
    /// Coarsen/block-aligned: stride == size.
    /// Sliding: stride < size.
    Window { size: usize, stride: usize },

    /// Sequential contiguous frames of variable widths.
    /// Example: resampling to a different frequency.
    Blocks { sizes: Vec<usize> },

    /// Compressed Blocks for quasi-periodic patterns.
    /// Example: monthly groupby on daily data — period = [31,28,…,31] × n_periods years.
    PeriodicBlocks { sizes: Vec<usize>, n_periods: usize },

    /// Arbitrary per-frame ranges, possibly overlapping.
    /// Example: rolling resample; np.ufunc.reduceat.
    VariableWindow { ranges: Vec<Range<usize>> },

    /// Group labels.
    /// Example: GroupBy; np.ufunc.at.
    Groups { labels: Vec<usize>, n_unique_labels: usize },
}

FramedArray

A FramedArray<F> is the array paired with its per-axis frames. The frame kind is a zero-size type parameter F — a sealed Framed marker (WholeAxis, Window, …) named to mirror the Frame variants — so frame-specific methods can be gated at the type level while the shared surface stays generic:

pub trait Framed: private::Sealed {
    const NAME: &'static str;
}

/// A frame over the whole axis. Mirrors [`Frame::WholeAxis`].
pub struct WholeAxis {}
/// A windowed frame (coarsen / block / sliding). Mirrors [`Frame::Window`].
pub struct Window {}

pub struct FramedArray<F: Framed> {
    expr: Arc<ArrayExpr>,
    view: HashMap<AxisName, Frame>,
    _framed: PhantomData<F>,
}

WholeAxis and Window are implemented today as Array::over and Array::coarsen, returning FramedArray<WholeAxis> and FramedArray<Window>. The aggregation surface (reduce, sum, mean) lives on impl<F: Framed> FramedArray<F>, so the same accumulator machinery serves plain reductions, coarsening, and rolling alike. Frame-specific methods (e.g. keepdims on WholeAxis) live on the narrower impl FramedArray<WholeAxis>.

Accumulators

Accumulators implement aggregations over a Frame in to a brix::State. For example, Array::over(["time"]).sum() records an ArrayOp::Reduce { acc, view } — the accumulator (SumAccumulator) plus the frame: view maps each reduced axis to a Frame (here {"time": Frame::WholeAxis { keepdims: false }}).

pub trait ArrayAccumulator: Debug + Send + Sync {
    /// Output dtype of this reduction.
    fn output_dtype(&self) -> DType;

    /// Output fill_value when reducing an input whose absent blocks all
    /// hold `input_fill`, over `n_elements` along the reduce axes. Used
    /// at plan time to give users immediate feedback on fill_value
    /// changes; not called at runtime.
    fn update_fill(&self, input_fill: Value, n_elements: u64) -> Value;

    // TODO: we might combine these two into AlgebraicProperties struct
    fn is_associative(&self) -> bool;
    fn is_commutative(&self) -> bool;

    /// Bytes of accumulator state per output element (used by the brix
    /// `cost` bridge in `LoweredAccumulator`).
    fn state_bytes_per_element(&self) -> u64;

    /// Allocate state shaped to one output brick.
    fn init(&self, shape: &[usize]) -> BrixResult<Box<dyn State>>;

    /// Fold one input frame's slot into `state` at the corresponding `state_frame_loc`.
    fn fold_frame(
        &self,
        // state to update
        state: &mut dyn State,
        // destination index in State to update
        // The accumulator is free to interpret that in the context of its own State.
        state_frame_loc: &HashMap<usize, usize>,
        // view to aggregate
        slot: ReadFrame<'_>,
    ) -> BrixResult<()>;

    /// Merge states (e.g. merge step of tree reduction)
    fn merge(&self, state: &mut dyn State, other: &dyn State) -> BrixResult<()>;

    /// Finalize output value from state (e.g. sum/count=mean)
    fn finalize(&self, state: Box<dyn State>, coord: &BrickCoordinate) -> BrixResult<Brick>;

    fn deserialize_state(&self, bytes: &[u8]) -> BrixResult<Box<dyn State>>;
}

An accumulator’s running state is a Box<dyn State> — opaque to the executor, which only passes it between accumulator methods and serializes it at stage boundaries. Each accumulator defines its own concrete struct (SumState, MeanState) and downcasts internally. Accumulator::State: Serialize is carried as a constraint for a possible future distributed execution and for possible incremental compute.

/// Opaque serializable state for accumulators (and other stateful ops).
///
/// Each accumulator defines its own concrete state struct (e.g., `SumState`,
/// `MeanState`) and downcasts internally. The executor never inspects state
/// — it only passes it between accumulator methods and serializes at stage
/// boundaries.
pub trait State: Debug + Send + Any {
    /// Serialize for inter-process exchange
    fn to_bytes(&self) -> BrixResult<Vec<u8>>;
    fn as_any(&self) -> &dyn Any;
    fn as_any_mut(&mut self) -> &mut dyn Any;
}

During lowering, each ArrayAccumulator is wrapped in a LoweredAccumulator adapter that implements the brix-layer brix::Accumulator trait — mirroring how OpaqueSource becomes LoweredOpaque:

#[derive(Debug, Clone)]
pub struct LoweredAccumulator {
    pub inner: Arc<dyn ArrayAccumulator>,
    // this is the ChunkGrid assuming keepdims=True
    // we squeeze out in finalize if keepdims is False.
    pub grid: ChunkGrid,
    pub view: HashMap<AxisName, Frame>,
}

It stamps the typed inner accumulator with the output grid and the view (the frame map), so the brix layer has everything it needs without knowing about frames or chunk sizes.

Folding over frames

Choices

  1. We initialize brix::State and accumulate into it, so ArrayAccumulator::fold_left takes &mut State. An ugly choice, but it matches what low-allocation array aggregations need.
  2. Aggregations within a brick are serial; we rely on cross-brick parallelism.

Here is how one brick’s worth of a reduction runs.

LoweredAccumulator::fold_left calls ArrayPayload::as_frame_iter(view) in a loop, and passes the returned value to ArrayAccumulator::fold_frame.

For each framed axis, the Frame implies a sequence of element-space ranges. The cartesian product of those per-axis sequences gives the frames we aggregate over. ArrayPayload::as_frame_iter(view) yields (state_frame_loc, framed_view) pairs, where state_frame_loc is the destination brix::State location to fold into.

Each framed_view is carved by the single per-backend primitive, slice_axes — given the ranges to slice out of the input ArrayPayload (axes not listed stay at full extent):

pub trait SliceFrames {
    fn slice_axes(&self, ranges: &[(usize, Range<usize>)]) -> ReadFrame<'_>;
}

Everything else on the frame-generation surface is range arithmetic over this one method, so extending to a new storage kind (e.g. sparse arrays) means adding one impl SliceFrames — the chunking and cartesian-product drivers are untouched.

The view is then handed to ArrayAccumulator::fold_frame(state, state_frame_loc, framed_view), which folds it into state at the lane named by state_frame_loc. The input slot is a ReadFrame a readonly-view:

#[derive(Debug, Clone)]
pub enum ReadFrame<'a> {
    Dense(TypedView<'a>),
    // Owned `FillArray` to reduce lifetime gymnastics.
    // This is cheap to Clone so it doesn't matter
    Fill(FillArray),
    // Future: Sparse(SparseData), Mixed(MixedData)
}

Lowering

Lowering is the single pass that turns the logical array IR into the array-agnostic Brix-IR (src/array/lower.rs). It walks the ArrayExpr post-order and, for each node:

  • walks the DAG and picks an element-sized chunk grid per node given resource constraints as an input.
  • bakes that grid into the concrete brix::Source / brix::Kernel / brix::Accumulator impls,
  • projects a counts-only brix::IterationSpace onto the resulting brix::Expr for the planner to consume.

All chunk assignment happens in this one place. Transparent sources (Fill, Dense, …) have no inherent chunk grid and can generate any brick shape on demand; opaque sources (Zarr) have native grids that lowering reads through. The brix layer never names ChunkGrid — the rich per-axis chunk representation stays entirely in array/.

The pass returns a Lowered:

#[derive(Clone)]
pub struct Lowered {
    pub expr: Arc<Expr>,
    // FIXME: This `root_grid` is a wart that exists only because the final sink
    // is not in the `expr`. It is passed separately, and so we need to carry the
    // explicit array-level chunk grid for that final Sink.
    pub root_grid: ChunkGrid,
    // This sidecar Map is simply to enable nice visualization/explainers.
    // It is *not* used for planning or execution at the brix layer.
    pub grids: HashMap<ExprId, ChunkGrid>,
}

expr is the Brix-IR DAG handed to the planner. root_grid and grids (per-node, keyed by brix::ExprId) retain the rich element-sized ChunkGrids — with their AxisChunks encoding and occupancy — that the counts-only brix::IterationSpace on each brix::Expr deliberately drops. Internal consumers (the renderer especially) read them back without re-running the lowering math; downstream brix execution never needs them.

Execution

The full pipeline — optimize → lower → plan → execute — is exercised end-to-end by the compute helper in tests/common/mod.rs:

pub fn compute<T: Element, A: ArrayLike + SinkFill>(
    execute: ExecuteFn,
    arr: &A,
    budget: usize,
) -> ndarray::ArrayD<T> {
    init_tracing();
    let optimized = arr.optimize().unwrap();
    let lowered = lower(&optimized, budget).unwrap();
    let physical = plan(&lowered.expr).unwrap();
    let _ = physical
        .nodes
        .iter()
        .find(|(_, info)| matches!(info, NodeInfo::Sink { .. }))
        .expect("plan must have a Sink node");
    let sink = InMemorySink::new(lowered.root_grid, arr.dtype(), arr.sink_fill()).unwrap();
    execute(&physical, budget, &sink).unwrap();
    sink.into_array::<T>()
}

The Lowering output’s expr is planned into a PhysicalPlan, an InMemorySink is sized from root_grid, and the chosen executor backend (execute is a function pointer so the same test body runs against every backend) streams bricks into the sink.

Brix

The brix layer is the array-agnostic Brix-IR (src/brix/). Think of it as a wiring diagram for routing bricks: it does not care about the contents of each brick, only where each brick needs to go.

A brix node is characterized purely by a brick grid — a shape with a brick count along each dimension, but no per-brick size.

Why the term brick?

We use brick deliberately to avoid confusion with chunks and shards as used in the storage layer (e.g. Zarr); tile is another common term. The brick grid is for orchestration — it need not be identical to, or even aligned with, the chunk or shard grid of the dataset (though there are benefits to aligning).

How a computation reaches and moves through this layer:

  • An optimization pass runs on the ArrayExpr (e.g. pushing indexing as low as possible).
  • A brick-size target is computed from the available execution resources.
  • The ArrayExpr is lowered to an Expr graph — the brix layer. Chosen chunk sizes are stamped onto each array Kernel that will be applied.
  • The Expr graph formally tracks only the number of bricks along each dimension of the IterationSpace; dtypes and actual chunk sizes ride along on the stamped kernels, not the graph.
  • A Schedule is written from the graph — an abstract plan annotated with loop ranges and opportunities for parallelism — then handed to an executor.

Memory budget

If a single brick exceeds the memory budget (e.g. a large Zarr chunk), this is a hard error for now rather than a silent compromise.

The brix layer must never import from the array layer; the dependency is one-way.

Concepts

The Brix layer exists because the fundamental operations for orchestrating array computations do not care about the “array-nature” of the work. So this layer is insulated from array-level concerns (dtypes, chunk sizes, etc.) and simply operates by routing abstract “bricks” to the right places.

One-way dependency

Code in array/ may depend on brix/, but brix/ must never import from array/. brix defines the interface, array/ provides the implementations.

Bricks

A brick is the unit of data routed through the brix layer (src/brix/brick.rs). Bricks are an orchestration concept: think of this layer as representing a wiring or circuit diagram for routing array chunks from producers to consumers.

Why the term brick?

We use brick deliberately to avoid confusion with chunks and shards as used in the storage layer (e.g. Zarr); tile is another common term.

Brick contents are described by the BrickPayload trait, which the array layer’s ArrayPayload (Dense, Fill, …) implements. The brix layer routes bricks without inspecting their contents — it only needs to know where each brick goes:

pub trait BrickPayload: Debug + Send + Sync + Any {

    /// Format is implementation-specific but conventionally tagged so a

Inputs

An edge in the DAG is an Input: a producer plus a per-axis description of how this consumer reads it — an “access relation” (term borrowed from polyhedral array compiler literature). An Op may have multiple Inputs.

/// An input to an expression node. Carries the producer `Arc<Expr>` plus a
/// per-axis [`AxisAccess`] describing how the consumer reads chunks from
/// this producer along that axis.
/// Axis transformations are either
///  - a `Composed` frame: planner applies these via `NamedLayout::compose`
///    when constructing a [`crate::brix::schedule::Schedule::Domain`] iteration space; or
///  - `Translated` per-coord override: imposed at runtime with [`crate::brix::schedule::AccessRelation::resolve_coord`].
#[derive(Debug, Clone)]
pub struct Input {
    pub expr: Arc<Expr>,
    pub access: HashMap<AxisName, AxisAccess>,
}

Access relations

Note

The vocabulary here gets complex and hard-to-read. Reading through the worked examples below will help.

Given the consumer brick coordinate for this iteration, we need to figure out which producer bricks to supply to the Op for each input. This relation is termed an "access relation".

Brix models this transformation as

(consumer iteration coordinate).pipe(compose with affine transform).pipe(non-affine translation)

along each axis. Thus Input::access maps each axis to an AxisAccess representing a transformation from producer IterationSpace to consumer IterationSpace:

pub enum AxisAccess {
    /// Affine transformations on this axis. The planner absorbs it into the
    /// access layout via `NamedLayout::compose`; nothing about this axis
    /// survives to runtime resolution.
    Composed(Frame),
    /// Per-coord override on this axis (slice offset, gather lookup,
    /// zipped gather followers). Carried through to runtime
    /// `resolve_coord` after the affine pass.
    Translated(TranslatedAccess),
    // Identity is assumed for any axis not in Input::access
}
  • Composed is an affine transformation (a Frame: whole-axis, window, …). The planner folds it into the producer iteration space through composition — it becomes part of the generated loop in the Schedule.
  • Translated are non-affine overrides (offsets for slicing, gather lookup, zipped gather followers) applied at runtime.
  • Any axis absent from the map is identity (pass-through / broadcast).

The translation per axis is an AxisOverride: a constant Offset (slicing) or a per-coord Lookup table (gather, i.e. orthogonal/vectorized indexing):

pub enum AxisOverride {
    /// Constant chunk-coord shift (slice): `c + k`.
    /// Affine, but kept as an override for symmetry with oindex & vindex.
    Offset(usize),
    /// Per-iteration lookup table (gather: oindex & vindex): `table[c]`.
    /// that assumes the keys are 0..len(Vec)
    /// TODO: eventually we'll likely need an inverted index to reverse this;
    ///       and thus a more complex type.
    Lookup(Vec<usize>),
}

A Translated axis carries a TranslatedAccess: either a single self-driven override (Independent — e.g. slicing or orthogonal indexing) or a co-iterated follower group (Zipped — e.g. vectorized indexing, where one produced axis drives gather tables on several consumed axes):

/// We need the ability to express iteration spaces as being composed in at
/// least two ways:
/// 1. the cartesian product of iteration spaces along each axis of the chunk grid
/// 2. a zipped iterator over iteration spaces of multiple axes of the chunk grid
///
/// For example, with the outer map key in `[brackets]`:
/// - Slicing: ["i" → Independent(AxisOverride::Offset(slice_start))]
/// - OIndex:  ["i" → Independent(AxisOverride::Lookup(outer_indices_for_i))]
/// - VIndex:  ["k" → Zipped({"i": idx_i, "j": idx_j})]
///   (key `k` = produced/leader axis; followers `i`,`j` are the consumed axes
///   co-iterated by `k`, each carrying its own gather table.)
#[derive(Clone, Debug)]
pub enum TranslatedAccess {
    /// Self-driven per-axis override: producer's chunk-coord on this
    /// axis is derived from this axis's own iter coord.
    /// Co-iterated follower group: producer's chunk-coord on each
    /// follower axis is derived from the leader axis's iter coord.
    Independent(AxisOverride),
    Zipped(HashMap<AxisName, AxisOverride>),
}

At the Schedule layer these resolve into a concrete AccessRelation (with its AffineAccess layout); the Brix-IR holds only the declarative per-axis AxisAccess form.

Iteration Space

The IterationSpace is the brix layer’s counts-only view of a node’s brick grid: the number of bricks along each dimension, with no per-brick size. Lowering projects the array layer’s ChunkGrid to an IterationSpace. The planner resolves iteration order and parallelism over this space; it does not pick chunk sizes.

pub struct IterationSpace {
    pub layout: NamedLayout,
    pub occupancy: Option<Occupancy>,
    pub limits: HashMap<AxisName, usize>,
}

The core data structure here is NamedLayout, an extension of CuTe Layouts that adds axis names. This structure was used to move quickly and because it provides convenient algebra (“logical divide”, “compose” etc). Honestly, our requirements are a bit more complicated (occupancy, non-affine transformations), so we may dump this structure in the future.

limits records per-axis bounds for axes that logical_divide has split into nested sub-modes (coarsen / block-windows). logical_divide(axis, w) rounds the tile count up, so the last block of bricks is smaller when the extent isn’t a multiple of w. The layout algebra can’t represent that ragged block, so the axis’s true extent is stashed here. IterationSpace::extent takes this in to account, and executors are expected to use that method at runtime to ensure the loops are correctly structured.

Sources & Sinks

The brix layer defines two array-agnostic seams for getting bricks into and out of a plan: Source produces brick payloads on demand, and Sink receives them as they are produced. Both are traits here — the concrete implementations live in the array layer and are baked into the IR at lowering.

What lives where

The brix layer never names a ChunkGrid or a dtype. A Source/Sink is addressed purely by BrickCoordinate; the planner reads only the per-axis brick counts off the surrounding Expr::space. Chunk sizes ride along, baked into the concrete impl as a private field.

Sources

A Source produces a brick payload for a requested chunk coord; the executor re-attaches the coord to build a Brick. The array-layer implementations (Fill/Dense/Zarr) are documented under Sources & Sinks.

pub trait Source: Debug + Send + Sync {
    fn get_brick_payload(&self, coord: &BrickCoordinate) -> BrixResult<Arc<dyn BrickPayload>>;
    fn cost(&self, coord: &BrickCoordinate) -> Cost;
}

Sinks

A Sink is the destination handed to execute(), outside the expression DAG. write_brick takes &self so writes can be concurrent; init/finalize bracket the run.

pub trait Sink: Send + Sync {
    fn init(&self) -> BrixResult<()>;
    fn write_brick(&self, brick: Brick) -> BrixResult<()>;
    fn finalize(&self) -> BrixResult<()>;
}

Schedule

Warning

This section is least thought-out and was sketched out with a serial for-loop executor in mind. Be critical when reading, and assume what semantics we might want for a good concurrent/parallel executor.

A Schedule (src/brix/schedule.rs) is the abstract plan of a computation emitted by the planner.

Conceptually, Brix considers all computations as effectively being loops over an iteration space. The Brix layer is an expression layer (“what to compute”) that gets lowered down to this execution layer (“how to compute”) Schedule. In principle, we could lower the Brix layer to many candidate Schedules.

The schedule is a tree of nested loops that controls iteration order, computation placement at loop levels, and stage fusion.

The Schedule is handed to an executor (src/executor/) for actual compute.

pub enum Schedule {
    /// Establish the iteration space for the enclosed body.
    /// `Loop` nodes inside look up their axis extents from here. One `Domain` per band;
    /// multi-band plans nest or sequence several.
    /// A distributed executor may construct Schedules for each worker by simply
    /// rewriting these Domain nodes to the appropriate subspace the worker is responsible for.
    Domain { iter_space: IterationSpace, body: Box<Schedule> },

    /// Iterate over a chunk dimension, identified by axis name.
    /// The range is looked up from the enclosing `Domain` at execute time.
    /// The enclosing `Domain`'s iteration space may be nested (e.g. for coarsen/rolling)
    /// So we include a `mode_index` to grab the "i-th mode" or "i-th sublayout"
    /// (Czeka 2026 Section 2.4.1) of the enclosing Domain's `iter_space`.
    Loop { axis: AxisName, mode_index: usize, body: Vec<Schedule> },

    /// Co-iterate `axes` over an explicit coordinate table
    /// (soon to be) filtered by the overarching Domain range (similar to Loop).
    /// `coords[i]` is the i-th present coord, length `axes.len()`,
    /// values in `axes` positional order.
    Zip { axes: Vec<AxisName>, coords: Arc<Vec<Vec<u32>>>, body: Vec<Schedule> },

    /// Execute a list of schedule steps in *specified* order at this loop level.
    /// Used to express top-level sequences (e.g. InitSink, loop nest,
    /// FinalizeSink).
    Sequence(Vec<Schedule>),

    /// Execute a list of schedule steps in *any* order at this loop level.
    /// Unused today
    // Set(Vec<Schedule>),

    /// Initialize state for a compute node that has core dimensions.
    /// Placed before the core-dim loops begin.
    ///
    /// Example, for Reduce, this creates accumulator state sized to the output brick.
    ///
    /// `access` is this node's iter→global access relation — used to derive the
    /// output coord that sizes the accumulator's initial state.
    Init { node: NodeName, access: AccessRelation },

    /// Per-iteration "compute" step. Examples:
    /// - Source: read brick at current coord into workspace.
    /// - Blockwise: read inputs from workspace, broadcast, apply op, store result.
    /// - Reduce: fold `workspace[input]` into accumulator.
    ///
    /// `inputs` names the workspace entries this step reads, in port order.
    /// Each input brick's coordinate was already fixed by the producer's own
    /// `Apply` (recompute model: the emission immediately precedes this read),
    /// so the edge carries no access of its own.
    /// The planner controls placement — a source Apply can be hoisted above
    /// inner loops for reuse when broadcasting is involved.
    ///
    /// `access` is this node's own iter→global translation (e.g. for the accumulator result).
    ///
    /// Sinks never appear in Apply — they have their own InitSink / WriteSink /
    /// FinalizeSink variants.
    Apply { node: NodeName, inputs: Vec<NodeName>, access: AccessRelation },

    /// Extract result from a compute node's state after core-dim iteration
    /// completes. Placed after the core-dim loops exit. Produces a brick
    /// in workspace for routing to the next step.
    ///
    /// For Reduce: calls accumulator.finalize() → workspace brick.
    Finalize { node: NodeName },

    /// FIXME: These sink-specific variants are a wart to me.
    /// Initialize a sink (allocate buffers, open files, etc.). Placed once,
    /// before any WriteSink into that sink executes.
    InitSink { sink: NodeName },

    /// Write `workspace[node]` into the sink at the current brick coord, and
    /// release `workspace[node]`. This is the explicit memory-release point
    /// for a brick on its way out of the system.
    ///
    /// `access` is this node's iter→global translation — used to derive the
    /// brick coord at which to write into the sink.
    WriteSink { node: NodeName, sink: NodeName, access: AccessRelation },

    /// Finalize a sink (commit writes, close files, etc.). Placed once,
    /// after every WriteSink into that sink has executed.
    FinalizeSink { sink: NodeName },
}

Cost

Before lowering, Brix computes a brick-size target for orchestration from the execution resources available, then plans against a memory budget. All terms in the memory model are known at plan time:

peak_memory = sum_sources(prefetch_capacity × brick_size)
            + recursion_stack (one brick per plan tree depth)
            + output_channel_capacity × output_brick_size
            + accumulator_states (for reduce nodes)

Bounded channels enforce backpressure at every I/O boundary.

Hard limits, not heuristics

If a single brick exceeds the budget (e.g. a large Zarr chunk), this is a hard error for now rather than a silent compromise. A future relaxation could read from the source multiple times with smaller slices.

The model is a flat budget of memory, I/O, and compute terms:

pub struct Cost {
    // --- Memory ---
    /// Output brick bytes resident in the workspace channel.
    pub output_bytes: u64,
    /// Persistent accumulator state (reduce nodes). Zero for non-stateful.
    pub state_bytes: u64,
    /// Transient workspace during apply (FFT twiddles, gather scratch, etc.).
    pub scratch_bytes: u64,

    // --- I/O ---
    /// Number of remote storage reads (dominates cloud cost/latency).
    pub io_read_requests: u64,
    /// Total bytes read from storage.
    pub io_read_nbytes: u64,
    /// Number of remote storage writes (spill, shuffle).
    pub io_write_requests: u64,
    /// Total bytes written to storage.
    pub io_write_nbytes: u64,

    // --- Compute ---
    pub flops: u64,
}

Algorithms

How the structural ops in the pipeline visualizer are implemented. Every section follows the same outline so the algorithms can be compared point-by-point:

  1. Expression & ArrayOp — what the user expression builds in the logical IR.
  2. Plan-time chunk-grid math — how array/lower.rs + array/chunk_grid/* construct the output ChunkGrid and the chunk-coord routing from output chunks back to input chunks.
  3. DAG embedding — the mid-IR Expr node lowering emits, and the access: HashMap<AxisName, AxisAccess> on its Input edge. An axis is either AxisAccess::Composed(Frame) — a structural transform the planner folds into the loop nest (NamedLayout::compose / logical_divide) — or AxisAccess::Translated(TranslatedAccess) — a per-coord override applied at runtime by AccessRelation::resolve_coord after the affine pass. Absence from the map is identity (pass-through / broadcast).
  4. Runtime (within-brick) — the element-space math the concrete Kernel / Accumulator does with one brick. Each impl bakes in both the input and output ChunkGrid plus the original indexer/view, so bounds are derived locally from one brick coordinate. All of them short-circuit Fill inputs — no element work.

Indexing

We treat indexing operations as Op::Blockwise nodes with specialized access relations for inputs. The complexity in how we model access relations (ADD LINK) is purely to support indexing semantics.

At a high-level, we subset the input bricks appropriately, and then apply a per-brick indexer appropriately. Below we work through three cases slicing (or basic indexing), orthogonal indexing, and vectorized indexing.

Slicing

Expression & ArrayOp

a.isel({"i": Indexer::Slicer(Slice::new(4, 12, 1))})

builds ArrayOp::Slice(HashMap<AxisName, Slice>) — one Slice { start, end, step } per indexed axis.

Plan-time chunk-grid math

Per sliced axis, each output chunk is the intersection of one source chunk with [start, step, end). For now, we are clever and encode this as an output chunk grid that is almost always AxisChunks::RLE along the subset axes. We also set chunk_offsets[axis] = k0, the source chunk that output chunk 0 maps to.

DAG embedding

Op::Blockwise(SliceKernel), with the edge carrying AxisAccess::Translated(TranslatedAccess::Independent(AxisOverride::Offset(k0))) for every input axis. At runtime AccessRelation::resolve_coord adds the offset to the affine-resolved producer coord. The override stack is ordered: two slices separated by a Blockwise (so fuse_adjacent_selects can’t collapse them) accumulate as stacked entries applied in planner descent order — Offset(1) then Offset(2) reads the producer at chunk c + 3 (the “stacked slices” workload).

Runtime (within-brick)

SliceKernel::clip derives element bounds from both grids’ slice_ranges, translates output positions through the slicer (start + pos·step) into source space, subtracts the input brick’s element start to get brick-local ranges, and slice_each_axises the dense payload. A Fill input short-circuits: re-emit a Fill payload at the output brick shape.

Not yet implemented

  • step != 1 — rejected at ArrayExpr::slice construction. The plan-time RLE math and SliceKernel::clip handle strides (unit-tested), but with step > chunk_size the skipped source chunks lower to zero-size output chunks that still fetch their source brick (the Offset mapping stays affine). Lifting this needs a richer access relation: a chunk-space stride when step is a multiple of the chunk size (purely affine), a logical_divide into a periodic two-mode layout in general — only the residue-nonlinear remainder needs a (period-sized) table. compose_slicers would also need step != 1 composition for fuse_adjacent_selects.
  • Slicing an AxisChunks::Variable axis — hard error in construct_sliced_output_chunk_grid.
  • Reversed slices: Slice fields are usize and step must be positive — no negative indices or negative step.

Orthogonal indexing

Expression & ArrayOp

a.isel({"i": Indexer::Outer(vec![1, 5]), "j": Indexer::Outer(vec![0, 3])})

builds ArrayOp::Oindex(HashMap<AxisName, Vec<usize>>) — one integer index vector per indexed axis, independent across axes (the output is their cartesian product).

Plan-time chunk-grid math

chunk_grid::oindex::output_grid_and_selection. Per indexed axis, OuterSelection::from_sorted_indices partitions the index vector into runs by source chunk: AxisChunks::locate resolves each index to (source_chunk, in_chunk_position), and consecutive indices hitting the same source chunk group into one AxisSelection { source_chunk, positions }. The source-chunk sequence must be non-decreasing — hard error otherwise — though within a chunk positions may be in any order. Indexing an AxisChunks::Variable axis is a hard error. The output axis becomes AxisChunks::Variable(run_sizes) — one output chunk per touched source chunk, so untouched source chunks are never read; unindexed axes pass through unchanged.

DAG embedding

Op::Blockwise(GatherKernel), with the edge carrying, per indexed axis, AxisAccess::Translated(TranslatedAccess::Independent(AxisOverride::Lookup(table))) where table[c] maps the axis’s output chunk index to its run’s source_chunk. Each axis’s Lookup is driven by that axis’s own iter coord (Independent), so the per-axis runs recombine as a cartesian product — this is the structural difference from vindex’s Zipped, where one leader axis drives every table.

Runtime (within-brick)

GatherKernel::gather: per indexed axis, slice the full index vector to output_bounds[axis] (the positions this output chunk owns), subtract the input brick’s element start to get brick-local positions, and select along the axis. A Fill input short-circuits to a Fill at the output brick shape.

Not yet implemented

  • Indexing an AxisChunks::Variable source axis — hard error.
  • Indices not sorted by source chunk — hard error, no auto-sort (a sort would be a shuffle; deliberately out of scope for now).
  • AxisOverride::Lookup is a forward table assuming dense 0..len keys; an inverted index for reversing the map is future work.

Vectorized indexing

Expression & ArrayOp

a.vindex(VectorizedIndexer::from_indices("k", indices)) builds ArrayOp::Vindex(VectorizedIndexer) — equal-length integer index vectors per consumed axis, zipped into a single produced axis k.

Plan-time chunk-grid math

chunk_grid::vindex::output_grid_and_selection, in two steps:

  • Run partition (VectorizedSelection::from_indexer): walk the k positions; AxisChunks::locate resolves each index into (source_chunk, in_chunk_position) per consumed axis. Consecutive positions sharing the same multi-axis chunk-coord tuple group into a run; the tuple sequence must be non-decreasing in lex order over the (sorted) consumed axes — hard error otherwise, no auto-sort. The result is VectorizedSelection { consumed_axes, per_axis } where per_axis[a][r] is an AxisSelection { source_chunk, positions } for axis a, run r.
  • Output grid: remove the consumed axes (without_axes) and append k as AxisChunks::Variable(run_widths) — one output chunk per run, so each output brick reads exactly one input brick.

DAG embedding

Op::Blockwise(VIndexKernel), with the edge access keyed by the produced axis, which does not exist on the producer grid: {"k" → AxisAccess::Translated(TranslatedAccess::Zipped({i: AxisOverride::Lookup(tᵢ), j: AxisOverride::Lookup(tⱼ)}))}. Each follower’s table maps run index → that axis’s source chunk. At runtime resolve_coord reads the leader k’s iter coord from the enclosing loop and drives every follower’s Lookup from it — the co-iteration that distinguishes vindex from oindex, whose per-axis TranslatedAccess::Independent(AxisOverride::Lookup) entries form a cartesian product.

Runtime (within-brick)

VIndexKernel::gather: for output run r, loop i over the run width, index_axis_move the source brick at per_axis[a][r].positions[i] on every consumed axis simultaneously, and assign the resulting cell into lane i of the produced axis. A Fill input short-circuits to a Fill at the output brick shape.

Not yet implemented

  • Index arrays must be plan-time literals — no lazy or computed index operands.
  • Indices not lex-sorted by input chunk coord — hard error, no auto-sort (see checkpoints/vindex.md; sorting is a shuffle).
  • A single produced axis only; VectorizedIndexer cannot produce a multi-dimensional output.
  • Fewer than two consumed axes is rejected (from_indices) — use orthogonal indexing instead.
  • AxisChunks::Variable consumed axes — hard error (mirrors oindex).

Reductions

Every reduction is the same ArrayOp::Reduce { acc, view } node — the accumulator says what to fold, the per-axis view (an element-space array::expr::Frame) says over which frames. Whole-axis mean and coarsen differ only in the Frame variant, and both ride the AxisAccess::Composed arm: the frame becomes loop structure (a planner-built fold loop), never a runtime coord override.

Whole-axis reduction (mean)

Expression & ArrayOp

a.over(&["i"]).mean() builds a FramedArray<WholeAxis> and then ArrayOp::Reduce { acc: MeanAccumulator, view: {"i": Frame::WholeAxis { keepdims: false }} }.

Plan-time chunk-grid math

ChunkGrid::after_reduce drops the axis from the output grid (keepdims: true keeps it as AxisChunks::Uniform { size: 1, n_chunks: 1 }). Lowering also builds a second grid with Frame::force_keepdims applied — the state grid, where every reduced axis survives at size 1 — and bakes it into the LoweredAccumulator, so intermediate state has a consistent shape regardless of the user’s keepdims.

DAG embedding

Op::Reduce(LoweredAccumulator), with the edge carrying AxisAccess::Composed(brix::Frame::WholeAxis { keepdims }) (Whole / Whole(keep) in the visualizer). The planner’s extend_iter_space_for_fold turns this into loop structure: with keepdims: false the axis is absent from the output iteration space, so it is appended flat and the whole depth-1 axis folds; with keepdims: true it logical_divides the axis by n_in into (1, n_in) and folds the inner mode. Either way the divide is exact — no predication limit (contrast coarsen below). The schedule wraps the fold loop with Init / Finalize nodes around the accumulator state.

Runtime (within-brick)

MeanState { sum, count } — the running sum stays in the input dtype, the count is I64. LoweredAccumulator::fold_left re-frames each incoming brick (as_frame_iter; for WholeAxis one slot spans the brick’s reduced extent) and calls fold_frame: sum-fold the slot, bump count by the framed volume (mean, not nanmean — every cell counts). A Fill slot contributes value × volume without densifying. merge adds both fields pairwise across parallel states. finalize promotes to f64 and divides sum / count — NumPy-faithful, mean over integers produces DType::F64 — then squeezes out axes whose view was WholeAxis { keepdims: false }.

Not yet implemented

  • Reducing along a sparse (occupancy-carrying) axis — hard error at lowering; needs the SparseAccumulator wrapper (V1).
  • nanmean / skip-NA semantics — fold_frame counts every cell.

Coarsen

Expression & ArrayOp

There is no coarsen op. a.coarsen({"i": 2}).sum() builds a FramedArray<Window> and then ArrayOp::Reduce { acc, view: {"i": Frame::Window { size: 2, stride: 2 }} } — the same framed reduce as above with a windowed Frame, still in element space.

Plan-time chunk-grid math

ChunkGrid::after_reduce coarsens the viewed axis’s chunks (AxisChunks::coarsen, extent ceil(n / size)) instead of dropping it. The state grid is the output grid itself — every output lane survives.

DAG embedding

Op::Reduce(LoweredAccumulator). build_reduce_access re-derives the element-space frame in chunk space. With w = ceil(n_in_chunks / n_out_chunks):

  • w == 1: the window fits inside one brick — identity edge (no map entry). The frame is invisible to the planner; all folding is element-space inside the accumulator.
  • w > 1: the window spans bricks — AxisAccess::Composed(brix::Frame::Window { size: w, stride: w }). The planner’s extend_iter_space_for_fold logical_divides the axis into (n_out, w) and appends a fold loop over the inner mode; a non-divisible n_in records a predication limit so the trailing partial window doesn’t read past the producer extent.

The two regimes are why the visualizer’s coarsen workload uses one-element bricks: it forces w = 2 so the Win(2,2) edge and the planner-generated fold loop become visible. With bricks at least as large as the window the same expression lowers to an identity edge.

Runtime (within-brick)

Identical machinery to the whole-axis case — only the framing differs. fold_left’s as_frame_iter cuts each brick into Window frames, and fold_frame is invoked once per (output lane, frame slot), so a window living inside one brick folds here rather than in the loop nest. finalize squeezes nothing: the coarsened axis survives on the output.

Not yet implemented

  • Sliding windows (stride != size) — hard error in both Frame::output_extent and the planner’s extend_iter_space_for_fold.
  • The other Frame variants — Blocks, PeriodicBlocks, VariableWindow, Groups — don’t lower yet (ChunkGrid::after_reduce and build_reduce_access error); the brix-side Frame::Ranges exists but nothing emits it.
  • Reducing along a sparse axis — same V0 restriction as above.
