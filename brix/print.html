The Brix Book

Motivation

Brix is an nD chunked-array computing system built in Rust.

It is intended to power analytic workflows classically driven by Xarray/Dask, bringing in ideas from OLAP databases, and also advance the state-of-the-art by innovating on interfaces and data models.

The problems

A few problems with the current Python data analysis stack from years of experience:

  • Low co-operation between compute and storage layers is a hard limiter.
    • DB systems deliver a lot of value through such co-operation: for example, “materialized views”, “incremental view maintenance”, and storing pre-computed statistics of the data to accelerate downstream queries. None of these valuable capabilities are meaningfully available to the scientific Python array computing community.
  • We leave a lot of performance on the table today.
    • Optimizing workloads is extremely hard. Because the code is divided between many independent OSS projects, a lot of context ends up being lost when communicating between layers. Good performance requires heroic engineering across multiple projects, some of which are more active than others.
  • The previous point is extremely important, now that we have Icechunk which has the ability to saturate network hardware in Rust. Actually doing so in python requires careful concurrency orchestration across Dask & Zarr. We need a compute stack that can take full advantage of Icechunk’s performance without relying on the user to be extremely knowledgeable.
  • Users are not isolated from the storage layer and are forced to consider “chunk sizes” at open-time. This is a major drag on performance, and user-friendliness.
  • Many datasets, particularly in Earth Sciences, are streaming (albeit relatively slow streams). For example, forecast and satellite data streams update at regular intervals updates roll in for forecasts. We want to support incremental, streaming compute in the future. Nothing in the Python stack today was designed with this in mind.
  • At its lowest level, the current python stack is a wrapper over C-level loops. This makes extensibility quite hard. The Numpy ufunc API is great e.g. (cumulative max =np.maximum.accumulate, grouped max = np.maximum.at) but fails at composing these primitives, requiring arcane workarounds (how do you run an cumulative max aggregation within groups?). Doing such complex operations usually involves using a compiled extension point (Numba/Rust/Pybind11 etc.). Such composition is required for many climate analytics workloads.

An ideal

  1. Predictable execution: either execute or fail hard if query cannot be satisfied with defined (mem/cpu/time etc.).
  2. Isolate the user from chunk-size choices.
  3. Easy execution next to the data: Separate execution from expression, allowing remote execution of the workload and return results. Following Halide, what to compute (the Expr graph) should be separated from how (the Schedule).
  4. Support input data from multiple data formats:
    1. At least Icechunk for array inputs
    2. Some tabular format for tabular inputs, including geometries for zonal statistics workloads.
  5. Friendy, readable, high-level interfaces: All interfaces should be designed so that AI can generate readable/skimmable/auditable code i.e. provide high-level primitives.
    1. Xarray-like: Rich interface that uses metadata and exposes n-dimensionality of the underlying data.
    2. SQL: must be able to accept queries expressed in SQL and return results in a tabular form
  6. Be as familiar as possible. To ease adoption, we will strive to provide an API surface familiar to existing Python data stack users, but will absolutely innovate on API.
  7. Usable from multiple languages : Python + Rust
  8. Support UDFs in both Rust and Python: key to supporting arbitrary user workloads
  9. Strong support for data schemas
  10. Export results to zero-copy standard interchange formats: DLPack + Arrow
    1. This is required so we can offer to_xarray or to_polars as an escape hatch
  11. Be able to saturate the machine
  12. Design around static shapes with explicit escape hatches. To start with, we assume that all extents are known at plan time. Data-dependent shapes (dropna, unique, boolean indexing) go through explicit materialization barriers. In practice, since Xarray requires static shapes, this isn’t a strong limiter on workloads.
  13. Explicit over implicit. Hard errors and limits over heuristics; user-controlled materialization hints over automatic CSE.
  14. Be amenable to developing incremental compute features in the future.
  15. Query graphs should be serializable to enable stored views.

Inspirations

The tabular data world has undergone a major transformation, with projects like Apache Arrow, Polars, DuckDB, Ibis, and many others innovating around performance and scalability of analytics tools. Apache DataFusion in particular has rolled up decades of accumulated knowledge about analytical query execution into a developer-friendly package. Overall these trends have favored:

  1. Compiled languages (Rust, C++) with multi-language bindings (Python, R, etc.)
  2. Performant multi-threaded single-node execution over complex distributed clusters
  3. SQL as a lingua franca for analytics

An equally vibrant array compiler / DSL space has also spun up in the last decade targeted at ML teams. MLIR/XLA/Triton/CuTile/”Tensor Comprehensions”/Halide/Mojo/Finch.jl/TACO/Chapel/Futhark/Accelerate all have the very similar goals - write once, execute optimally on different hardware. These projects are unifying 50 years of HPC work on matrix computation (e.g. BLAS) with more generic array transformations needed for ML kernels.

Architecture

Brix is a layered nD chunked-array compute orchestration system.

Goals

  1. Separate expression from execution allowing saving the expression graph to disk.
  2. Expose composable primitives.
  3. Preserve as much context as possible about patterns in the data. This enables both expression graph optimizations and automatic choosing of chunk sizes for orchestration.
  4. Intelligently choose chunk sizes for orchestration through cost-based planning. Account for storage layout, expression graph, and knowledge about the data participating in the workload.
  5. Be generic enough to support multiple in-memory data formats reflecting the variety of data used in the wild (dense arrays, sparse arrays, Arrow geometries, etc.).
  6. First class support for the Zarr data model (fill_value, write_empty_chunks=False) including an explicit representation of “sparsely populated” chunk grids of data on-disk.
  7. Expose extension points. The goal is to provide an expression layer as well as a parallel execution layer; not to innovate on in-memory array storage by duplicating ndarray (for example).

Concepts

At a high level there are three layers:

  • the array layer characterized by shape, dtype, chunk grid with chunk sizes, and dimension names
  • the “brix” layer but with arrays abstracted away) characterized purely by a “brick grid” with a shape but no per-brick size.
  • the “schedule” layer that translates a brix-level DAG to an iterator-level description consumed by an executor.

Think of the “array layer” as expression “what to compute” in array-element space (e.g. rolling window over 5 elements along axis 'x').

Think of the “brix layer” as representing a wiring diagram for routing chunks bricks; to do so we do not care about the contents of each brick, just that we know where to route it. This is again “what to compute” but with actual array contents abstracted away.

Think of the “schedule layer” as expressing “how to compute” with limited degrees of freedom exposed to the executor.

Each layer carries strictly less semantic information than the one above it.

Note

We are using the term “brick” intentionally to avoid confusion with “chunks” and “shards” as used in the storage layer (e.g. Zarr). “Tile” is another term used commonly.

The brix grid (IterationSpace) is for orchestration, it need not be identical to or even aligned with the “chunk” or “shard” grid of the dataset (though of course there are benefits to doing so).

We distinguish between

  1. Expression-time: A user calls a method on Array.
  2. Submit-time: A user submits the compute graph to the server.
  3. Plan-time: The system walks through the submitted expression graph, optimizes the graph, plans explicitly sized chunk grids for executing each expression node, and lowers to a brix graph. The brix graph is then converted to the PhysicalPlan
  4. Run-time: The executor receives the PhysicalPlan and executes it. At this time, ArrayPayloads are routed through to the right kernels for computation.

Choices

Three-layer separation

Choosing the array+brix+iteration layer is a choice.

Separating the Array from Brix (Iteration Space/chunk grid operations) has some benefits. It makes it clear that the core orchestration is manipulating arbitrary data payloads over a iteration space (‘brix grid’) that only cares about the Cost per brick rather than details (dtype/chunk size/ etc.) This leaves us open to orchestrate other arbitrary data payloads in the future, for example orchestrating a grid of filenames for (virtual) ingest. As prior art, Xarray uses dask as a pure task scheduling engine precisely at ingest time, where manipulating a list of filenames in parallel is necessary to construct a data cube.

An implication that has both down-side and up-side is that the same concept is commonly expressed at both the Array layer and the Brix layer (e.g. sliding windows/slicing). This can get confusing. The up-side though is that it’s easy to spot when we (really agents) mix array-layer concerns in to the brix-layer. Personally, I have found it far easier to reason about the code once I added the separation.

Choosing to have an iteration layer / Schedule is a choice that comes from realizing that in most array workloads we have a lot of information available at plan-time (see below).

One-way door

The choice to split between array & brix is a one-way door (mostly). It would be a massive refactor to merge the two layers but not too hard with Claude. I belive the separation makes the codebase easier to understand, and Claude code easier to audit.

The choice to add an iteration layer is a two-way door (below).

Static shapes

An extremely important choice is whether concrete shapes (axis extents) are known at plan-time. Concrete shapes help us choose concrete brick grids / iteration spaces at lowering-time. This limitation applies to Xarray today. Dask has limited ability to support arrays with “unknown” shapes (only known at compute-time), mostly so as to convert from DataFrames of unknown size.

Unknown shapes are mostly useful only with 1D input and less so with nD. For e.g. a[a > 0] with numpy returns a 1D array regardless of the dimensionality of a. In general, such a filter operation in nD generally results in a ragged array which most array packages can’t represent so they return a flattened 1D array as a result.

One way to think of this is as conceptualizing arrays are bounded or unbounded streams of chunks. When we know shapes at plan-time, we can do random access within streams, partition them etc.

Similarly the same issue pops up on array.unique(). This one is interesting. To find all unique values you must look through all bricks (using a tree-reduction say). You don’t know how many unique values exist a-priori, so the output is unknown-size. If array is of known shape, then you do know how many bricks you have to process. This is still a reduction over a bounded stream.

Another use case where unknown shapes pop-up is when grouping by an out-of-core array (example). If you don’t know number of unique groups then the result (groupby(large_array).mean()) has a dimension of unknown size. That said, there’s almost always sidecar metadata that will specify number of unique groups (at least) and the group labels. Again we could just insert a barrier that calculates the unique groups (and perhaps their distribution) and uses that to plan the following stages.

One-way door

Supporting unknown shapes would be a pretty major change to execution.

A place this may bite us is we are trying to join against or convert from tables of unknown extent e.g. a filtered dataframe. We could require a materialization barrier if we wanted to support this workflow nicely.

Execution Model

Push vs pull vs interpreter

We are choosing to have the Executor be an interpreter of the Schedule.

DB query engines express pipelines through a runtime protocol between operators: pull (Volcano next()) or push (consume()). Both approaches converge to nearly the same thing. (Shaikhha et al. 2016).

In the case of arrays, shapes, chunk grids, and iteration spaces are known at plan-time. We need not need to rely on heuristics (e.g. cardinality) that a DB executor does. So the planner emits the fused loop nest directly. Co-iteration of multiple inputs — which forces pull engines to coordinate demand and push engines to buffer one side — is resolved at plan-time and recorded using access relations along the edge. This kind of co-iteration is important for broadcasting e.g. A[i,j] + B[j]. At plan-time, we can hoist B’s read above the i loop and reuse the brick across iterations.

I don’t understand how DBs could achieve such an optimization given that they are geared toward data-dependent joins. Here’s what Claude has to say:

DB engines achieve the same reuse, but indirectly: no operator knows which loop level it sits at, so plans insert materialization instead (Postgres Materialize/Memoize, a hash join’s build side) — caching B wholesale and paying lookup cost, rather than holding one brick at the right loop depth. That said we could probably get similar behaviour by inserting appropriate “hints”.

This kind of imperative “schedule” is common to array compilers, most clearly Halide, whose algorithm/schedule split TVM and TACO inherit. Tensor Comprehensions makes the same separation in polyhedral form — its schedules are the ISL schedule trees cited in Planner. Chapel’s domain maps are similar: they factor data distribution and iteration out of the loop body.

Two-way door

The choice to add an iteration layer (Schedule) is a two-way door. I am currently considering the Schedule to be the interface to the executor. But this need not be true. We could pass an Arc<Expr> graph directly, and have the executor decide how to run it.

Further, one might interpret today’s version of the Schedule as a very opinionated and annotated suggestions for how to execute things. The executor is free to re-interpret that information as it wishes.

Data Model

Arrays ([ArrayLike]) are a logical wrapper around expressions ([ArrayExpr]) that represent streams of [ArrayPayload]. Arrays are a thin wrapper on top of [ArrayExpr] presenting a more familiar, broader API surface to the user.

All arrays are characterized by:

  1. expr - An expression ([ArrayExpr]) representing a graph of operations that create this array.
  2. dtype - A data type ([DType])
  3. shape - a tuple of ints, one per axis.
  4. dims - Dimension names per axis

This contract is enforced with the [ArrayLike] trait — one required method, expr(), with the rest defaulting to read through that expression :

/// Trait for user-facing array variants ([`Array`], [`BlockSparseArray`]).
/// Required method: [`Self::expr`].
/// Everything else has a default that reads through the expr.
pub trait ArrayLike: std::fmt::Debug + Send + Sync {
    /// The array-level IR node this array wraps.
    fn expr(&self) -> &Arc<ArrayExpr>;

    fn shape(&self) -> Vec<usize> {
        self.expr().shape()
    }

    fn dtype(&self) -> DType {
        self.expr().dtype.clone()
    }

    fn dims(&self) -> Vec<&str> {
        self.expr().dims().iter().map(|s| s.as_str()).collect()
    }

    /// Run array-level optimization passes.
    fn optimize(&self) -> ArrayResult<Array> {
        Ok(Array::from(optimize::optimize(self.expr())?))
    }

    /// Fill value for absent / unwritten output positions —
    /// This is a _logical_ fill value for sparse arrays.
    /// So it is `Some` for sparse outputs; `None` for dense outputs.
    /// TODO: this seems like a (small) wart
    fn sink_fill(&self) -> Option<Value>;

    /// Wrap this array in a terminal [`ArraySink`], yielding an `Array` ready
    /// for `lower`/`plan`/`execute`. Fallible: the sink validates the output
    /// dtype (e.g. a fill value of the wrong dtype) at construction.
    fn to_sink(&self, sink: Arc<dyn ArraySink>) -> ArrayResult<Array> {
        Ok(Array::from(Arc::new(ArrayExpr::sink(self.expr().clone(), sink)?)))
    }

    /// Materialize this (lazy) array and re-wrap the result as a fresh
    /// in-memory `Array`. Runs the full pipeline — optimize → sink → lower →
    /// plan → execute — into an [`InMemorySink`], then feeds the assembled
    /// payload back through [`Array::from_payload`]. `memory_budget` bounds peak
    /// memory during execution (see [`crate::lower::lower`]).
    ///
    /// Dtype-agnostic: dense and extension payloads round-trip identically.
    fn load(&self, executor: impl Executor, memory_budget: usize) -> ArrayResult<Array>
    where
        Self: Sized,
    {
        let rooted = self
            .optimize()?
            .to_sink(Arc::new(InMemorySink::new(self.sink_fill().map(Into::into))))?;
        let physical = plan(&lower(&rooted, memory_budget)?.expr)?;
        let payload = executor
            .execute(&physical)?
            .into_values()
            .next()
            .expect("in-memory sink always yields exactly one output");
        Ok(Array::from_payload(ArrayPayload::from_payload(&*payload).clone(), &self.dims()))
    }

    /// The backing in-memory payload, when this array is a bare transparent
    /// [`InMemorySource`] — i.e. one produced by [`Array::from_payload`] or
    /// materialized by [`Self::load`]. `None` for any other expression
    /// (constant fills, opaque/IO sources, or arrays carrying unevaluated ops).
    fn payload(&self) -> Option<&ArrayPayload> {
        match &self.expr().op {
            ArrayOp::TransparentSource(src) => {
                src.as_any().downcast_ref::<InMemorySource>().map(|s| &s.0)
            }
            _ => None,
        }
    }
}

Arrays are lazy

Cheap metadata manipulation — that of dtype, shape, dims — is reflected immediately as feedback to the user. All actual computation of values is deferred.

Arrays have no chunks at expression-time.

Important

Arrays have no chunk sizes (or chunk grids) associated with them!

At lowering time, an explicit [ChunkGrid] is chosen for each Array node in the graph based on cost considerations. This [ChunkGrid] translates to a Brix [IterationSpace].

Arrays

We include two concrete arrays that impl ArrayLike motivated by Zarr’s storage model where fill_value chunks need not be stored on disk. We decompose to two axes:

  1. a dense or sparse chunk grid (“iteration space”);
  2. dense / sparse array (whether fill_value is a semantically valid value that requires propagation).

For example, consider adding 1 to an array backed by a Zarr OpaqueSource where some on-disk chunks are missing / interpreted as all fill-value. If the fill_value is semantically invalid (e.g. NaN) then:

  1. we needn’t do anything at all for fill-value chunks.
  2. we add 1 to every populated chunk. If the fill_value is semantically valid (e.g. 0) then
  3. we simply need to update the fill_value to 0+1=1.
  4. we add 1 to every populated chunk.

We model:

  1. the underlying chunk grid as dense/sparse (translates to a dense/sparse iteration space at the brix layer) using brix::Occupancy in both ChunkGrid and brix::IterationSpace.
  2. model the presence of a valid fill_value through a BlockSparseArray that adds a new fill value property to Array.

Choice

BlockSparseArray requires explicit fill_value treatment and propagation much in exactly the same way that sparse matrices do. Thus it makes sense to split that up from arrays where the fill_value is invalid and no such logic is required so I chose to not use fill_value: Option<Value> on Array.

The Array IR

Array is a thin handle — the operations you call on it are recorded into ArrayExpr: the array-layer IR (array/src/expr.rs).

An ArrayExpr node is its op plus its inputs and the metadata that makes the layer labelled and dtype-aware.

/// Array-level expression node.
#[derive(Debug, Clone)]
pub struct ArrayExpr {
    pub op: ArrayOp,
    pub inputs: Vec<Arc<ArrayExpr>>,
    // array shape, strides, order
    pub layout: ArrayLayout,
    pub dtype: DType,
    // currently unused, but anticipating usefulness of distinguishing
    // sparse/dense arrays for planning/optimizer purposes
    pub brick_kind: ArrayPayloadKind,
    pub label: Option<String>,
}

Each node wraps an ArrayOp. The op set is deliberately explicit — separate Unary/Binary rather than a single general Map — while the optimizer’s needs are still being worked out:

/// Array-level op.
/// These are pretty explicit at the moment anticipating that matching
/// against specific variants are useful for optimization.
/// i.e. Unary, Binary are separated out, and not just a general Map.
#[derive(Clone)]
pub enum ArrayOp {
    // Transparent sources are _not_ chunked and any chunk grid may be imposed.
    // E.g. in-memory arrays and functional arrays (fill, future arange/eye).
    TransparentSource(Arc<dyn TransparentSource>),
    // Opaque sources are chunked and have a chunk preference
    // E.g. Zarr/Icechunk
    OpaqueSource(Arc<dyn OpaqueSource>),
    // For generic external scalar operations, e.g. many string operations
    // that may be provided by zax-arrow
    // Elemwise(Arc<dyn MapKernel>)
    Literal(ScalarValue),
    Unary(UnaryOp),
    /// Elementwise binary op — always two inputs. A scalar operand
    /// (`x + 1.0`) enters as a rank-0 [`ArrayOp::Literal`] input,
    Binary(BinaryOp),
    Reduce {
        acc: Arc<dyn ArrayAccumulator>,
        /// Frame applied to each reduce axis
        view: HashMap<AxisName, Frame>,
    },
    /// Per-axis positional slicing. One `Slice` per indexed axis.
    Slice(HashMap<AxisName, Slice>),
    /// Per-axis orthogonal integer indexing.
    Oindex(HashMap<AxisName, Vec<usize>>),
    /// Lined-axis vectorized integer indexing.
    Vindex(VectorizedIndexer),
    /// Cast to a target dtype. Either inserted at plan-time for implicit NumPy-style
    /// promotion (see [`ArrayExpr::binary`]) or user-specified by `Array::cast`.
    AsType(DType),
    Sink(Arc<dyn ArraySink>),
    // Future: Transpose, Reshape, ...
}

The bricks that flow along the graph’s edges at execution time are Payloads.

Payloads

Array payloads have three axes, how they are bundled is up for debate.

  • Storage: dense/sparse/extension
  • Validity mask
  • DataType: primitive/extension

The ops describe the graph; what flows along its edges at execution time is a stream of bricks. The contents of an array-layer brick are an ArrayPayload:

/// Array-layer brick payload. The arms are the closed *kind* tag; the data
/// behind each arm answers the uniform [`ArrayData`] contract.
// Keep this in sync with [`ArrayPayloadKind`], [`WriteFrame`]
#[derive(Debug, Clone)]
pub enum ArrayPayload {
    Dense(DenseArray),
    /// Externally provided payload (datetime, categorical, string, geometry).
    Extension(Arc<dyn ArrayData>),
    // Sparse(Arc<dyn SparseArray>)
}

ArrayPayload impls brix::BrickPayload so brix never inspects these variants (see Traits).

Choice

We choose to model Dense ndarray payloads and some future sparse payload explicitly. Everything else (including complex logical types like datetimes, categoricals, strings are Extension with an ExtensionDType). We do so because

  • dense ndarrays of primitive types are the most common payload,
  • we need to maintain a cross-dtype promotion rules for various primitive dtypes, and
  • we need first-class support for mixed sparse-dense workloads (tensordot for regridding).

Dense array

DType/Value describe a single element; a brick’s worth of dense values is held in a DenseArray — a dtype-tagged ArcArrayD<T>. This is the dense container only; extension kinds carry their own representation behind ArrayData. ArcArray storage makes clones and slices zero-copy buffer shares, so sliced frames are owned payloads with no view lifetimes; a consumer that knows its concrete T borrows an ArrayViewD<T> via DenseArray::as_typed.

/// Dense-kind value buffer, tagged by dtype. This is the dense container only —
/// `ArcArrayD<T>` per primitive type T.
///
/// `ArcArray` storage makes clones and [`Index::slice`] outputs zero-copy
/// buffer shares.
///
/// FIXME: Sadly mutation ([`Element::array_mut_from_dense`]) is copy-on-write;
/// the writers (accumulator state, sink buffer) hold the sole
/// reference by construction, so the copy never fires.
#[derive(Debug, Clone)]
pub enum DenseArray {
    F64(ArcArrayD<f64>),
    I64(ArcArrayD<i64>),
}

Ops traits

Every payload kind — core’s DenseArray and externally provided extension arrays alike — implements the same two traits: Index for per-brick indexing and ArrayData for the remaining structural contract (dtype, shape, squeeze, sink writes). ArrayPayload delegates to them uniformly.

/// Per-brick structural indexing. Every method returns an owned
/// [`ArrayPayload`]; `slice` is zero-copy by kind contract (`Dense` shares
/// the parent buffer through `ArcArray`, `Extension` impls buffer-share
/// Arrow-style), while `oindex`/`vindex` gather into fresh buffers.
pub trait Index {
    // TODO: we need an iindex - "integer index" too
    fn slice(&self, slicers: &[(usize, Slice)]) -> ArrayPayload;
    fn oindex(&self, indexers: &[(usize, Vec<usize>)]) -> ArrayPayload;
    fn vindex(&self, indexers: &[(usize, Vec<usize>)], produces: usize) -> ArrayPayload;
}
/// The uniform contract over payload data — what every payload
/// kind supports regardless of storage representation.
/// TODO: break this up in to more super-traits when it makes sense.
///       this will let us be more explicit about the contract for each use-case
///       e.g. ArrayData is too much for SinkBuffer;
///       Note: traits cant have constructors, so `empty` is not here but on the DType.
///       DType is the only thing the LoweredSink holds.
pub trait ArrayData: Index + Send + Sync + Debug {
    /// Logical element dtype of this payload.
    fn dtype(&self) -> DType;

    /// Per-axis element extents.
    fn shape(&self) -> &[usize];

    /// Drop the given size-1 axes (ascending positions; extents pre-validated
    /// by the caller, [`ArrayPayload::squeeze`]). A buffer-sharing reshape.
    fn squeeze(&self, axes: &[usize]) -> BrixResult<ArrayPayload>;

    /// Write `data` into the element-space region `ranges` (one per axis) —
    /// the sink population path, only ever called on a buffer the sink
    /// exclusively owns (core-allocated, or via
    /// [`crate::extension::ExtensionDType::empty`]). Accept the payloads
    /// this data kind can ingest; reject the rest.
    fn write_range(&mut self, ranges: &[Range<usize>], data: &ArrayPayload) -> BrixResult<()>;

    /// The kernel-side downcast hook.
    fn as_any(&self) -> &dyn Any;
}

Data Types

Every array carries a DType, and scalars are dtype-tagged values. We distinguish between opaque and interpreted variants:

#[derive(Debug, Clone)]
pub enum DType {
    Primitive(PrimitiveType),
    Extension(Arc<dyn ExtensionDType>),
}
  • Primitive — those with a Rust equivalent (i64, f64). Core owns these because numeric promotion needs a closed set of rules implemented.
  • Extension — everything not primitive: datetime, categorical, string, geometry. Fully external, behind Arc<dyn ExtensionDType>.

PrimitiveType and promotion

#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum PrimitiveType {
    F64,
    I64,
    // Future: F32, I32, U8, Bool.
}
  • PrimitiveType::promote implements the single symmetric promotion ruleset, following the Array API type-promotion rules. It is deliberately closed over DType::Primitive. Extension dtypes have no promotion lattice with primitives — binary ops involving them are delegated to the external crate.
  • A mixed-dtype binary op (e.g. int + float) inserts a plan-time AsType node (see The Array IR) so unary op kernels never see mixed inputs.
  • Scalar operands follow weak promotion (promote_weak), the Array API’s mixing arrays and Python scalars rule: a scalar contributes only its kind (integral vs. floating), never its width — i64_arr + 2 stays i64, i64_arr * 2.5 promotes to f64.
  • Per-op output dtypes (e.g. mean → f64) live on the ops, not in the lattice.

Scalars: Value and ScalarValue

Again we have a primitive-vs-extension split:

/// Typed scalar.
#[derive(Debug, Clone, Copy, PartialEq)]

    F64(f64),
    I64(i64),
}
/// Host-side typed scalar across both dtype arms — [`Value`] is the
/// primitive-only tagged union; this adds the extension arm. The dtype is
/// derived from the value (mirroring `ArrayPayload::dtype`), not stored.

    Primitive(Value),

Extensions

The Extension arm — the dtype, host scalar, and payload contracts a satellite crate implements to plug in a new logical dtype — is covered in Extensions.

Extensions

This page describes the mechanism by which a satellite crate (zax-time, zax-geo, …) plugs a new logical dtype. Three contracts need to be implemented — a dtype, a scalar, and an array payload.

An extension dtype is identity plus plan-time capability gates — no compute methods. Runtime compute dispatches through the paired payload, never the dtype.

/// An externally-owned logical dtype.
///
/// Carries **no compute methods**: a dtype is identity plus plan-time
/// opt-in capability gates (e.g. "i support binary ops").
/// Carries allocation primitives, since a DType is the only concrete
/// handle we have at runtime. ArrayPayload::empty() cannot be implemented
/// because Rust.
pub trait ExtensionDType: DynEqHash + Send + Sync + Debug {
    /// Identity / display name (e.g. `"datetime64[ns]"`).
    fn name(&self) -> &str;

    /// Storage element size in bytes — an estimate for variable-size types.
    /// Feeds the same cost/chunk-budget math as `PrimitiveType::itemsize`.
    fn itemsize(&self) -> usize;

    /// The binary-op capability, or `None` if this dtype advertises no
    /// binary ops (then every `binary()` involving it is a plan-time error).
    /// FIXME: we may need to be smarter here, but for now... this is OK.
    fn binary(&self) -> Option<&dyn BinaryCapability>;

    /// Allocate a writable host array of `shape` with unspecified contents —
    /// dtype-driven construction (`np.empty(shape, dtype)`). Lives on the
    /// dtype because no array exists yet to dispatch through; sinks populate
    /// the result via [`ArrayData::write_range`].
    fn empty(&self, shape: &[usize]) -> Box<dyn ArrayData>;

    /// Allocate a writable host array of `shape` filled with `fill` —
    /// `np.full(shape, fill, dtype)`. Infallible like [`Self::empty`]:
    /// `fill` belonging to this dtype is a caller-validated precondition,
    /// so the satellite downcast is a genuine invariant (`expect`). Serves
    /// sink prefill (extension fills) and rank-0 literal materialization.
    fn full(&self, fill: &dyn ExtensionScalar, shape: &[usize]) -> Box<dyn ArrayData>;
}

empty/full live on the dtype (not the array) because they construct from nothing — Array API empty(shape, dtype) — and sinks need them before any array exists.

Binary Ops

Binary ops dispatch through a capability modeled directly on Python’s __add__/__radd__ operand-coercion protocol. Because planning precedes materialization, BinaryCapability::apply runs with no brick: it infers the output dtype, inserts operand casts, and returns a built ArrayExpr.

/// Build `this <op> other` when this dtype owns the op. The direct analogue
/// of Python's `__add__`/`__radd__` operand-coercion protocol;
/// `apply` runs with no brick — it infers the output dtype, inserts operand casts,
/// and returns a built [`ArrayExpr`].
///
/// Operands are [`Operand`]s, so scalars dispatch through the same protocol
/// (an extension-*scalar* lhs dispatches via its dtype).
///
/// Contract, mirroring Python exactly
/// (<https://docs.python.org/3/reference/datamodel.html#emulating-numeric-types>):
/// - `None`           ≙ `__op__` returned `NotImplemented` → try the other operand
/// - `Some(Ok(expr))` ≙ a handled `__op__`/`__rop__` — the full lowered subgraph
/// - `Some(Err(e))`   ≙ the op is mine, but the operands are invalid (a real
///   error, not a decline)
pub trait BinaryCapability {
    fn apply(
        &self,
        op: BinaryOp,
        this: &Operand,
        other: &Operand,
    ) -> Option<ArrayResult<ArrayExpr>>;
}

An extension crate is expected to handle both array and scalar operands.

    /// by the caller, [`ArrayPayload::squeeze`]). A buffer-sharing reshape.

    /// Write `data` into the element-space region `ranges` (one per axis) —
    /// the sink population path, only ever called on a buffer the sink
    /// exclusively owns (core-allocated, or via
    /// [`crate::extension::ExtensionDType::empty`]). Accept the payloads
    /// this data kind can ingest; reject the rest.

    fn as_any(&self) -> &dyn Any;
}

So an extension array implements, in order of how much it can lean on Arrow-style buffer sharing:

  • Structural indexing — Index - slice/oindex/vindex.
  • Shape manipulation — squeeze (drop size-1 axes).
  • Metadata — dtype and shape.
  • The sink path — write_range, called only on a buffer the sink exclusively owns; the impl accepts the payloads its kind can ingest and rejects the rest.
  • The kernel hook — as_any, so a satellite kernel can downcast back to its concrete array and compute on it.

Example: a minutes dtype

This toy minutes dtype stores an i64 count per element. A full satellite spells out three structures — the dtype, the scalar, and the array payload — plus the binary capability that ties them together.

The dtype

The dtype is identity (name, itemsize) plus the capability gates and the empty/full constructors. No compute lives here — it dispatches through the capabilities and the payload:

/// Toy satellite dtype: an i64 count of minutes. `minutes ± minutes` and
/// `minutes * i64` are mine (built over the i64 storage, retagged);
/// everything else declines.
#[derive(Debug, PartialEq, Eq, Hash)]
struct Minutes;

impl Minutes {
    fn dtype() -> DType {
        DType::Extension(Arc::new(Minutes))
    }
}

impl ExtensionDType for Minutes {
    fn name(&self) -> &str {
        "minutes"
    }
    fn itemsize(&self) -> usize {
        8
    }
    fn binary(&self) -> Option<&dyn BinaryCapability> {
        Some(self)
    }
    fn empty(&self, shape: &[usize]) -> Box<dyn ArrayData> {
        Box::new(MinutesArray(ArrayD::zeros(IxDyn(shape))))
    }
    fn full(&self, fill: &dyn ExtensionScalar, shape: &[usize]) -> Box<dyn ArrayData> {
        let m = fill
            .as_any()
            .downcast_ref::<MinutesScalar>()

The array payload

Its payload wraps an ArrayD<i64> and implements the ArrayData contract.

/// Toy satellite payload: minutes stored as `ArrayD<i64>`. A real impl
/// buffer-shares on `slice` (Arrow-style); this test impl copies.
#[derive(Debug)]
struct MinutesArray(ArrayD<i64>);

impl Index for MinutesArray {
    fn slice(&self, slicers: &[(usize, Slice)]) -> ArrayPayload {
        let v = slicers.iter().fold(self.0.view(), |mut v, (ax, s)| {
            v.slice_axis_inplace(
                Axis(*ax),
                ndarray::Slice::new(s.start as isize, Some(s.end as isize), s.step as isize),
            );
            v
        });
        ArrayPayload::Extension(Arc::new(MinutesArray(v.to_owned())))
    }

    fn oindex(&self, indexers: &[(usize, Vec<usize>)]) -> ArrayPayload {
        let out = indexers
            .iter()
            .fold(self.0.clone(), |out, (axis, positions)| out.select(Axis(*axis), positions));
        ArrayPayload::Extension(Arc::new(MinutesArray(out)))
    }

    fn vindex(&self, _indexers: &[(usize, Vec<usize>)], _produces: usize) -> ArrayPayload {
        unimplemented!("not exercised by these tests")
    }
}

impl ArrayData for MinutesArray {
    fn dtype(&self) -> DType {
        Minutes::dtype()
    }
    fn shape(&self) -> &[usize] {
        self.0.shape()
    }
    fn squeeze(&self, axes: &[usize]) -> BrixResult<ArrayPayload> {
        let v = axes.iter().rev().fold(self.0.view(), |v, &p| v.index_axis_move(Axis(p), 0));
        Ok(ArrayPayload::Extension(Arc::new(MinutesArray(v.to_owned()))))
    }
    fn write_range(&mut self, ranges: &[Range<usize>], data: &ArrayPayload) -> BrixResult<()> {
        let src = match data {
            ArrayPayload::Extension(e) => e
                .downcast_ref::<MinutesArray>()
                .ok_or(BrixError::DowncastError { expected: "MinutesArray" })?,
            _ => {
                panic!("minutes sink accepts minutes bricks only");
            }
        };
        let mut view = self.0.view_mut();
        for (axis, range) in ranges.iter().enumerate() {
            view.slice_axis_inplace(
                Axis(axis),
                ndarray::Slice::from(range.start as isize..range.end as isize),
            );
        }
        view.assign(&src.0);
        Ok(())
    }
    fn as_any(&self) -> &dyn Any {
        self
    }
}

as_any is the hook that lets a satellite kernel recover the concrete MinutesArray from the dyn ArrayData it was handed. The kernel selected for the minutes dtype calls payload.downcast_ref::<MinutesArray>() (which routes through as_any) to get back its i64 buffer.

Binary ops

The toy minutes dtype owns minutes ± minutes and minutes * i64, and declines everything else.

Since the underlying storage is ArrayD<i64> this code reuses the internal i64 path. Note the use of ArrayError::ExtensionDowncastError for nice error handling.

fn retag(e: &Arc<ArrayExpr>, dtype: DType) -> Arc<ArrayExpr> {
    Arc::new(ArrayExpr { dtype, ..e.as_ref().clone() })
}

impl BinaryCapability for Minutes {
    /// Supports minutes ± minutes and minutes * i64, and declines everything else.
    fn apply(
        &self,
        op: BinaryOp,
        this: &Operand,
        other: &Operand,
    ) -> Option<ArrayResult<ArrayExpr>> {
        use BinaryOp::*;

        // Can we handle the requested operation?
        let other_dtype = other.dtype();
        let handled = match op {
            // minutes ± minutes
            Add | Sub => other_dtype.as_extension().is_some_and(|e| e.is::<Minutes>()),
            // minutes * i64
            Mul => matches!(other_dtype, DType::Primitive(PrimitiveType::I64)),
        };
        if !handled {
            // don't know what to do here, bail.
            return None;
        }

        // Resolve operands to their i64 storage and reuse the core's binary op path.
        let as_i64 = |o: &Operand| -> ArrayResult<Operand> {
            Ok(match o {
                Operand::Expr(e) => Operand::Expr(retag(e, DType::I64)),
                Operand::Scalar(ScalarValue::Primitive(v)) => Operand::from(*v),
                Operand::Scalar(ScalarValue::Extension(s)) => {
                    // A scalar that advertises the minutes dtype but isn't a
                    // `MinutesScalar` is a malformed operand
                    let m = s
                        .downcast_ref::<MinutesScalar>()
                        .ok_or(ArrayError::ExtensionDowncastError { expected: "MinutesScalar" })?;
                    Operand::from(Value::I64(m.0))
                }
            })
        };
        Some((|| -> ArrayResult<ArrayExpr> {
            let expr = ArrayExpr::binary(op, as_i64(this)?, as_i64(other)?)?;
            Ok(ArrayExpr { dtype: Minutes::dtype(), ..expr })
        })())
    }
}

Executing end-to-end

With the dtype and payload in hand, a minutes array runs through the full pipeline like any built-in dtype.

Sources & Sinks

Sources

The array layer has two flavors of source, split on whether the data has an inherent chunking.

A transparent source has no inherent chunk grid — lowering may impose any brick shape on demand:

/// Transparent source — generates any requested brick on demand, cheaply
/// and infallibly. The trait behind functional arrays: an impl is a generation rule
/// (constant, range, function) plus its shape and dtype.
///
/// `read` is infallible by contract — validation happens at expression-time.
/// For now, fallibe sources should be [`OpaqueSource`].
/// So out-of-range (for example) is detected at plan-time
/// For the moment, there is no cost hook in favor of `numel × itemsize`
/// from grid + dtype.
pub trait TransparentSource: Debug + Send + Sync {
    /// Per-axis element extents.
    fn shape(&self) -> &[usize];

    /// Element dtype this source produces. Sources are an entry point for
    /// dtype info the same way they are for shape.
    fn dtype(&self) -> DType;

    /// Plan-time promise of the payload kind `read` emits. Explicit, not
    /// derived from dtype — kind is orthogonal to dtype (a transparent
    /// `eye(n)` could emit Sparse bricks of primitive dtype).
    fn brick_kind(&self) -> ArrayPayloadKind;

    /// Generate the brick covering `bounds` (element space, one range per
    /// axis).
    fn read(&self, bounds: &[Range<usize>]) -> ArrayPayload;

    /// Short display label (viz, debug output).
    fn label(&self) -> String;

    /// The downcast hook (viz data extraction, satellite kernels).
    fn as_any(&self) -> &dyn Any;
}

An opaque source (Zarr-like) owns its native chunk grid which may carry occupancy for block-sparse data:

/// Opaque source — I/O-backed (Zarr, Arrow, etc.).
/// The optimizer can't see through this; it only knows shape, layout, and
/// `ArrayPayloadKind`. The source owns its chunk grid (data is already chunked
/// on disk).
///
/// Lowering wraps an `Arc<dyn OpaqueSource>` in a [`LoweredOpaque`]
/// before handing it to the planner.
pub trait OpaqueSource: Debug + Send + Sync {
    fn shape(&self) -> &[usize];
    /// The source's chunk grid — opaque sources own theirs (already
    /// chunked on disk / in storage). Lowering reads it to project a
    /// `brix::IterationSpace` and bake it into the [`LoweredOpaque`] wrapper.
    fn native_grid(&self) -> &SourceNativeChunkGrid;
    // Grab data for this range in array index space.
    fn read(&self, bounds: &[Range<usize>]) -> BrixResult<Arc<dyn BrickPayload>>;
    fn cost(&self, coord: &BrickCoordinate) -> Cost;

    /// Element dtype the source produces.
    fn dtype(&self) -> DType;

    fn brick_kind(&self) -> ArrayPayloadKind;
}

Lowering wraps either flavor in a grid-carrying adapter — [LoweredTransparent] / LoweredOpaque — which includes the chosen ChunkGrid and impls the brix::Source trait. The transparent impls today are FillSource (constant functional array, emits materialized dense bricks) and [InMemorySource] (any in-memory ArrayPayload, the entry point for extension data). Note that the chosen ChunkGrid need not align with the OpaqueSource’s native chunk grid.

Aggregations

Aggregations are (at the moment) reductions over one or more axes — sum, mean, and friends. They are backed by accumulators and computed over a frame.

extern crate brix_array;
use std::collections::HashMap;
use brix_array::{Array, ArrayError};

fn main() -> Result<(), ArrayError> {
let data = Array::fill(1.0, vec![3, 5], &["time", "x"]);
// Reduce over the whole `time` axis.
let totals = data.clone().over(&["time"])?.sum()?;
// Coarsen `x` into width-2 blocks; 5 isn't a multiple of 2, so the last block is a width-1 tail.
let coarsened = data.coarsen(HashMap::from([("x", 2)]))?.mean()?;
Ok(())
}

Frames

A frame is a lens over an array: a way of viewing it as a stream of windows rather than as a flat block (array/src/framed.rs). The framing model borrows from two places:

  1. numpy ufunc methods — reduce collapses a whole axis (WholeAxis), reduceat reduces over segment boundaries (Blocks/VariableWindow), and at scatters into indexed groups (Groups). A Frame is the descriptor that picks which behavior the same accumulator runs. Unlike ufuncs, our intent is to allow scans (“cumulative ops”, accumulate) to compose with these Frames.
  2. SQL:2003 window functions — an aggregate evaluated against a moving frame rather than the whole input, spelled as an OVER clause (SUM(x) OVER (...)) and optionally factored out as a named WINDOW W AS (...). Brix’s over plays the role of OVER, and a single frame model subsumes both halves of the clause: PARTITION BY grouping is Groups (per-element labels), while the frame spec is WholeAxis, Window, or Blocks.

All of these are just frame shapes fed to the same accumulator machinery.

Warning

Only WholeAxis and Coarsen have been “implemented” so far.

Choice

These enums are quite specific (in keeping with “preserve context as much as possible”). Consider though that we can use this information to smartly choose variable chunk sizes (e.g. chunks that line up with PeriodicBlocks makes it an embarassingly parallel problem).

/// This enum enables our approach to composing Aggregators (Reduce, Scan) with
/// different _views_ or _frames_ along an axis
/// (`rolling`, `groupby`, `coarsen`, etc.)
#[derive(Clone, Debug, PartialEq, Eq)]
pub enum Frame {
    /// Reduce the whole axis; preserve size-1 axis if keepdims is true
    /// TODO: We could remove keepdims; and insert a Squeeze if necessary?
    WholeAxis { keepdims: bool },

    /// Regular contiguous frames.
    /// Coarsen/block-aligned: stride == size.
    /// Sliding: stride < size.
    Window { size: usize, stride: usize },

    /// Sequential contiguous frames of variable widths.
    /// Example: resampling to a different frequency.
    Blocks { sizes: Vec<usize> },

    /// Compressed Blocks for quasi-periodic patterns.
    /// Example: monthly groupby on daily data — period = [31,28,…,31] × n_periods years.
    PeriodicBlocks { sizes: Vec<usize>, n_periods: usize },

    /// Arbitrary per-frame ranges, possibly overlapping.
    /// Example: rolling resample; np.ufunc.reduceat.
    VariableWindow { ranges: Vec<Range<usize>> },

    /// Group labels.
    /// Example: GroupBy; np.ufunc.at.
    Groups { labels: Vec<usize>, n_unique_labels: usize },
}

FramedArray

A FramedArray<F> is the array paired with its per-axis frames. The frame kind is a zero-size type parameter F — a sealed Framed marker (WholeAxis, Window, …) named to mirror the Frame variants — so frame-specific methods can be gated at the type level while the shared surface stays generic:

pub trait Framed: private::Sealed {
    const NAME: &'static str;
}

/// A frame over the whole axis. Mirrors [`Frame::WholeAxis`].
pub struct WholeAxis {}
/// A windowed frame (coarsen / block / sliding). Mirrors [`Frame::Window`].
pub struct Window {}

pub struct FramedArray<F: Framed> {
    expr: Arc<ArrayExpr>,
    view: HashMap<AxisName, Frame>,
    _framed: PhantomData<F>,
}

WholeAxis and Window are implemented today as Array::over and Array::coarsen, returning FramedArray<WholeAxis> and FramedArray<Window>. The aggregation surface (reduce, sum, mean) lives on impl<F: Framed> FramedArray<F>, so the same accumulator machinery serves plain reductions, coarsening, and rolling alike. Frame-specific methods (e.g. keepdims on WholeAxis) live on the narrower impl FramedArray<WholeAxis>.

Accumulators

Accumulators implement aggregations over a Frame in to a brix::State. For example, Array::over(["time"]).sum() records an ArrayOp::Reduce { acc, view } — the accumulator (SumAccumulator) plus the frame: view maps each reduced axis to a Frame (here {"time": Frame::WholeAxis { keepdims: false }}).

pub trait ArrayAccumulator: Debug + Send + Sync {
    /// Output dtype of this reduction.
    fn output_dtype(&self) -> DType;

    /// Output fill_value when reducing an input whose absent blocks all
    /// hold `input_fill`, over `n_elements` along the reduce axes. Used
    /// at plan time to give users immediate feedback on fill_value
    /// changes; not called at runtime.
    fn update_fill(&self, input_fill: Value, n_elements: u64) -> Value;

    // TODO: we might combine these two into AlgebraicProperties struct
    fn is_associative(&self) -> bool;
    fn is_commutative(&self) -> bool;

    /// Bytes of accumulator state per output element (used by the brix
    /// `cost` bridge in `LoweredAccumulator`).
    fn state_bytes_per_element(&self) -> u64;

    /// Allocate state shaped to one output brick.
    fn init(&self, shape: &[usize]) -> BrixResult<Box<dyn State>>;

    /// Fold one input frame's slot into `state` at the corresponding `state_frame_loc`.
    fn fold_frame(
        &self,
        // state to update
        // For now, this mut is OK because we expect to be parallelizing over loop dimensions,
        // constructing one State per output brick, and folding in to it.
        state: &mut dyn State,
        // destination index in State to update
        // The accumulator is free to interpret that in the context of its own State.
        state_frame_loc: &HashMap<usize, usize>,
        // frame to aggregate (an owned ideally-zero-copy slice of the input brick)
        input: ArrayPayload,
    ) -> BrixResult<()>;

    /// Merge states (e.g. merge step of tree reduction)
    fn merge(&self, state: &mut dyn State, other: &dyn State) -> BrixResult<()>;

    /// Finalize output value from state (e.g. sum/count=mean)
    fn finalize(&self, state: Box<dyn State>, coord: &BrickCoordinate) -> BrixResult<Brick>;

    fn deserialize_state(&self, bytes: &[u8]) -> BrixResult<Box<dyn State>>;
}

An accumulator’s running state is a Box<dyn State> — opaque to the executor, which only passes it between accumulator methods and serializes it at stage boundaries. Each accumulator defines its own concrete struct (SumState, MeanState) and downcasts internally. Accumulator::State: Serialize is carried as a constraint for a possible future distributed execution and for possible incremental compute.

/// Opaque serializable state for accumulators (and other stateful ops).
///
/// Each accumulator defines its own concrete state struct (e.g., `SumState`,
/// `MeanState`) and downcasts internally. The executor never inspects state
/// — it only passes it between accumulator methods and serializes at stage
/// boundaries.
pub trait State: Debug + Send + Any {
    /// Serialize for inter-process exchange
    fn to_bytes(&self) -> BrixResult<Vec<u8>>;
    fn as_any(&self) -> &dyn Any;
    fn as_any_mut(&mut self) -> &mut dyn Any;
}

During lowering, each ArrayAccumulator is wrapped in a LoweredAccumulator adapter that implements the brix-layer brix::Accumulator trait — mirroring how OpaqueSource becomes LoweredOpaque:

It stamps the typed inner accumulator with the output grid and the view (the frame map), so the brix layer has everything it needs without knowing about frames or chunk sizes.

Folding over frames

Choices

  1. We initialize brix::State per output brick and accumulate into it, so ArrayAccumulator::fold_left takes &mut State. An ugly choice, but it reduces memory allocations.
  2. Aggregations within a brick are serial; we rely on cross-brick parallelism.

Here is how one output brick’s of a reduction is constructed. Consider one brick of data of shape [time=3, x=5] — and run coarsen({"x": 2}).mean(). The view is {"x": Window { size: 2, stride: 2 }}; time is absent, so it stays a full-extent loop axis.

For each framed axis, the Frame implies a sequence of element-space ranges. The cartesian product of those per-axis sequences gives the frames we aggregate over. For our example:

state_frame_locframed_view (Index::slice)slot shape
{x: 0}time ∈ 0..3, x ∈ 0..2[3, 2]
{x: 1}time ∈ 0..3, x ∈ 2..4[3, 2]
{x: 2}time ∈ 0..3, x ∈ 4..5[3, 1]

Now ArrayPayload::as_frame_iter(view) yields (state_frame_loc, framed_view) pairs, where state_frame_loc is the destination brix::State location to fold into. Each framed_view is created by using [Index::slice], reusing the existing slicing machinery.

LoweredAccumulator::fold_left calls ArrayPayload::as_frame_iter(view) in a loop, and passes the returned view to ArrayAccumulator::fold_frame, which folds it into state at the location named by state_frame_loc.

The input to be aggregated is passed in as an owned ArrayPayload; ideally this is a zero-copy slice of the input brick. Dense shares the brick’s buffer through ArcArray; ideally Extension implement implement similar buffer sharing.

Lowering

Warning

The selection of chunk sizes, and detection/handling of “pipelines” is not implemented.

The actual lowering and application of kernels is implemented.

Lowering is the single pass that turns the logical array IR into the array-agnostic Brix-IR (array/src/lower.rs). It walks the ArrayExpr post-order and, for each node:

  • walks the DAG and picks an element-sized chunk grid per node given resource constraints as an input.
  • bakes that grid into the concrete brix::Source / brix::Kernel / brix::Accumulator impls,
  • projects a counts-only brix::IterationSpace onto the resulting brix::Expr for the planner to consume.

All chunk assignment happens in this one place. Transparent sources (FillSource, InMemorySource, …) have no inherent chunk grid and can generate any brick shape on demand; opaque sources (Zarr) have native grids that lowering reads through. The brix layer never names ChunkGrid — the rich per-axis chunk representation stays entirely in array/.

The pass returns a Lowered:

#[derive(Clone)]
pub struct Lowered {
    pub expr: Arc<Expr>,
    // This sidecar Map is simply to enable nice visualization/explainers.
    // It is *not* used for planning or execution at the brix layer.
    pub grids: HashMap<ExprId, ChunkGrid>,
}

expr is the Brix-IR DAG handed to the Brix planner.

Brix

Concepts

The brix layer is the array-agnostic Brix-IR (brix/src/). Think of it as a wiring diagram for routing bricks: it does not care about the contents of each brick, only where each brick needs to go.

The Brix layer exists because the fundamental operations for orchestrating array computations do not care about the “array-nature” of the work. So this layer is insulated from array-level concerns (dtypes, chunk sizes, etc.) and simply operates by routing abstract “bricks” to the right places.

The idea is to record the ‘wiring diagram’ parametrically so we can reason about it, both for execution, as well as for eventual incremental computation.

One-way dependency

Code in array/ may depend on brix/, but brix/ must never import from array/. brix defines the interface, array/ provides the implementations.

Bricks

A brick is the unit of data routed through the brix layer (brix/src/brick.rs). Bricks are an orchestration concept: think of this layer as representing a wiring or circuit diagram for routing array chunks from producers to consumers.

Why the term brick?

We use brick deliberately to avoid confusion with chunks and shards as used in the storage layer (e.g. Zarr); tile is another common term.

Brick contents are described by the BrickPayload trait, which the array layer’s ArrayPayload (Dense, Fill, …) implements. The brix layer routes bricks without inspecting their contents — it only needs to know where each brick goes:

pub trait BrickPayload: Debug + Send + Sync + Any {
    /// Serialize for inter-process exchange (shared storage, DBSP checkpoint).
    ///
    /// Forward-looking interface only — no concrete format is committed yet.
    fn to_bytes(&self) -> BrixResult<Vec<u8>>;

    fn as_any(&self) -> &dyn Any;
}

Operations

Wiring diagrams are constructed as a sequence of brick-level operations Op.

/// The operation performed by an expression node.
#[derive(Clone)]
pub enum Op {
    Source(Arc<dyn Source>),
    Blockwise(Arc<dyn Kernel>),
    Reduce(Arc<dyn Accumulator>),
    Sink(Arc<dyn Sink>),
    // Scan(Arc<dyn Scanner>)
}

The vocabulary here is simpler than that at the ArrayOp layer, and is the familiar data-manipulation vocabulary : map/reduce/scan/etc. These operations manipulate BrickPayloads on an IterationSpace. ‘Iteration Spaces’ are chunk grids abstracted down to purely number of chunks per axis and an ‘occupancy’ that describes whether that chunk needs to be processed.

Inputs

An edge in the DAG is an Input: a producer plus a per-axis description of how this consumer reads it — an “access relation” (term borrowed from polyhedral array compiler literature). An Op may have multiple Inputs.

/// An input to an expression node. Carries the producer `Arc<Expr>` plus a
/// per-axis [`AxisAccess`] describing how the consumer reads chunks from
/// this producer along that axis.
/// Axis transformations are either
///  - a `Composed` frame: planner applies these via `NamedLayout::compose`
///    when constructing a [`crate::schedule::Schedule::Domain`] iteration space; or
///  - `Translated` per-coord override: imposed at runtime with [`crate::schedule::AccessRelation::resolve_coord`].
#[derive(Debug, Clone)]
pub struct Input {
    pub expr: Arc<Expr>,
    pub access: HashMap<AxisName, AxisAccess>,
}

Access relations

Note

The vocabulary here gets complex and hard-to-read. Reading through the worked examples below will help.

Given the consumer brick coordinate for this iteration, we need to figure out which producer bricks to supply to the Op for each input. This relation is termed an "access relation".

Brix models this transformation as

(consumer iteration coordinate).pipe(compose with affine transform).pipe(non-affine translation)

along each axis. Thus Input::access maps each axis to an AxisAccess representing a transformation from producer IterationSpace to consumer IterationSpace:

pub enum AxisAccess {
    /// Affine transformations on this axis. The planner absorbs it into the
    /// access layout via `NamedLayout::compose`; nothing about this axis
    /// survives to runtime resolution.
    Composed(Frame),
    /// Per-coord override on this axis (slice offset, gather lookup,
    /// zipped gather followers). Carried through to runtime
    /// `resolve_coord` after the affine pass.
    Translated(TranslatedAccess),
    // Identity is assumed for any axis not in Input::access
}
  • Composed is an affine transformation (a Frame: whole-axis, window, …). The planner folds it into the producer iteration space through composition — it becomes part of the generated loop in the Schedule.
  • Translated are non-affine overrides (offsets for slicing, gather lookup, zipped gather followers) applied at runtime.
  • Any axis absent from the map is identity (pass-through / broadcast).

The translation per axis is an AxisOverride: a constant Offset (slicing) or a per-coord Lookup table (gather, i.e. orthogonal/vectorized indexing):

pub enum AxisOverride {
    /// Constant chunk-coord shift (slice): `c + k`.
    /// Affine, but kept as an override for symmetry with oindex & vindex.
    Offset(usize),
    /// Per-iteration lookup table (gather: oindex & vindex): `table[c]`.
    /// that assumes the keys are 0..len(Vec)
    /// TODO: eventually we'll likely need an inverted index to reverse this;
    ///       and thus a more complex type.
    Lookup(Vec<usize>),
}

A Translated axis carries a TranslatedAccess: either a single self-driven override (Independent — e.g. slicing or orthogonal indexing) or a co-iterated follower group (Zipped — e.g. vectorized indexing, where one produced axis drives gather tables on several consumed axes):

/// We need the ability to express iteration spaces as being composed in at
/// least two ways:
/// 1. the cartesian product of iteration spaces along each axis of the chunk grid
/// 2. a zipped iterator over iteration spaces of multiple axes of the chunk grid
///
/// For example, with the outer map key in `[brackets]`:
/// - Slicing: ["i" → Independent(AxisOverride::Offset(slice_start))]
/// - OIndex:  ["i" → Independent(AxisOverride::Lookup(outer_indices_for_i))]
/// - VIndex:  ["k" → Zipped({"i": idx_i, "j": idx_j})]
///   (key `k` = produced/leader axis; followers `i`,`j` are the consumed axes
///   co-iterated by `k`, each carrying its own gather table.)
///
/// As a side-note, neither Mullin's Mathematics of Arrays calculus, nor CuTe layouts,
/// provide any formalism for oindex/vindex operations. Those two harmonize the affine
/// access pieces, with MoA apparently folding in the AxisOverride::Offset piece too.
#[derive(Clone, Debug)]
pub enum TranslatedAccess {
    /// Self-driven per-axis override: producer's chunk-coord on this
    /// axis is derived from this axis's own iter coord.
    /// Co-iterated follower group: producer's chunk-coord on each
    /// follower axis is derived from the leader axis's iter coord.
    Independent(AxisOverride),
    Zipped(HashMap<AxisName, AxisOverride>),
}

At the Schedule layer these resolve into a concrete AccessRelation (with its AffineAccess layout); the Brix-IR holds only the declarative per-axis AxisAccess form.

Analogs

  1. Polyhedral model — Source of the term “access relation”. A map from a statement’s iteration domain to the array elements it touches, as in ISL/Polly.
  2. MLIR — Composed is an affine map: linalg structured ops use indexing_maps (AffineMaps) to relate the iteration space to each operand’s element access.
  3. IREE — Translated is the non-affine escape hatch. IREE’s LinalgExt dialect adds gather, scatter, sort, topk — ops that don’t fit linalg’s affine-indexing model. Our AxisOverride::Lookup (gather) is the same move: pull the irregular access out of the affine layer and carry it as a runtime table.

Iteration Space

The IterationSpace is the brix layer’s counts-only view of a node’s brick grid: the number of bricks along each dimension, with no per-brick size. Lowering projects the array layer’s ChunkGrid to an IterationSpace. The planner resolves iteration order and parallelism over this space; it does not pick chunk sizes.

pub struct IterationSpace {
    pub layout: NamedLayout,
    pub occupancy: Option<Occupancy>,
    pub limits: HashMap<AxisName, usize>,
}

Analogs

  1. Chapel — a domain defines both an iteration space and the shape of the arrays declared over it; IterationSpace plays the same role over the brick grid.
  2. CuTe — layouts Layout as a representation of nested loops.
  3. Halide — iteration domains with bounds inference.
  4. Dex — index set. The difference is that Dex’s index set is a static type; while IterationSpace is a runtime value projected during lowering.

Named Layouts

The core data structure here is NamedLayout, an extension of CuTe Layouts that adds axis names. This structure was used to move quickly and because it provides convenient algebra (“logical divide”, “compose” etc). Chapel provides similar operations over their Domains. Honestly, our requirements are a bit more complicated (occupancy, non-affine transformations), so we may dump this structure in the future.

A layout is CuTe’s shape : stride pair per mode. A mode is one dimension of a layout. The shape gives each mode’s extent, the stride the step it takes in the mapped space. Together they map a coordinate (one index per mode) to a single integer offset — the inner product of the coordinate with the strides (e.g. (3,4):(4,1) is a row-major 3×4 grid). So far this is the NumPy style shape/stride model. Now the difference: shapes and strides are hierarchical integer tuples, so any mode can itself nest a tuple of sub-modes. E.g. zarr chunks are nested with zarr shards. A flat axis has a single mode — its extent. logical_divide subdivides a mode into nested sub-modes, which we address as axis.k, the (axis, mode_index) pair:

  • Before: i is a flat axis of 6 bricks: one mode, written i (extent 6).
  • After: logical_divide(i, 2) (coarsen i by 2) — i splits into two sub-modes: i.0 is the surviving outer mode (ceil(6/2) = 3, keeps the axis name) and i.1 is the folded tile/window (2). The axis is still named i; it now owns modes i.0 and i.1.

A NamedLayout attaches an axis name to each top-level mode.

Limit override

limits records per-axis bounds for axes that logical_divide has split into nested sub-modes (coarsen / block-windows). logical_divide(axis, w) rounds the tile count up, so the last block of bricks is smaller when the extent isn’t a multiple of w. The layout algebra can’t represent that ragged block, so the axis’s true extent is stashed here. IterationSpace::extent takes this in to account, and executors are expected to use that method at runtime to ensure the loops are correctly structured.

Sources & Sinks

The brix layer defines two array-agnostic seams for getting bricks into and out of a plan: Source produces brick payloads on demand, and Sink receives them as they are produced. Both are traits here — the concrete implementations live in the array layer and are baked into the IR at lowering.

The brix layer never names a ChunkGrid or a dtype. A Source/Sink is addressed purely by BrickCoordinate; the planner reads only the per-axis brick counts off the surrounding Expr::space. Chunk sizes ride along, baked into the concrete impl as a private field.

Schedule

Warning

This section is least thought-out and was sketched out with a serial for-loop executor in mind. Be critical when reading, and assume what semantics we might want for a good concurrent/parallel executor.

A Schedule (brix/src/schedule.rs) is the abstract plan of a computation emitted by the planner.

Conceptually, Brix considers all computations as effectively being loops over an iteration space. The Brix layer is an expression layer (“what to compute”) that gets lowered down to this execution layer (“how to compute”) Schedule. In principle, we could lower the Brix layer to many candidate Schedules.

The schedule is a tree of nested loops that controls iteration order, computation placement at loop levels, and stage fusion.

The Schedule is handed to an executor (brix/src/executor/) for actual compute.

pub enum Schedule {
    /// Establish the iteration space for the enclosed body.
    /// `Loop` nodes inside look up their axis extents from here. One `Domain` per band;
    /// multi-band plans nest or sequence several.
    /// A distributed executor may construct Schedules for each worker by simply
    /// rewriting these Domain nodes to the appropriate subspace the worker is responsible for.
    Domain { iter_space: IterationSpace, body: Box<Schedule> },

    /// Iterate over a chunk dimension, identified by axis name.
    /// The range is looked up from the enclosing `Domain` at execute time.
    /// The enclosing `Domain`'s iteration space may be nested (e.g. for coarsen/rolling)
    /// So we include a `mode_index` to grab the "i-th mode" or "i-th sublayout"
    /// (Czeka 2026 Section 2.4.1) of the enclosing Domain's `iter_space`.
    Loop { axis: AxisName, mode_index: usize, body: Vec<Schedule> },

    /// Co-iterate `axes` over an explicit coordinate table
    /// (soon to be) filtered by the overarching Domain range (similar to Loop).
    /// `coords[i]` is the i-th present coord, length `axes.len()`,
    /// values in `axes` positional order.
    Zip { axes: Vec<AxisName>, coords: Arc<Vec<Vec<u32>>>, body: Vec<Schedule> },

    /// Execute a list of schedule steps in *specified* order at this loop level.
    /// Used to express top-level sequences (e.g. InitSink, loop nest,
    /// FinalizeSink).
    Sequence(Vec<Schedule>),

    /// Execute a list of schedule steps in *any* order at this loop level.
    /// Unused today
    // Set(Vec<Schedule>),

    /// Initialize state for a compute node that has core dimensions.
    /// Placed before the core-dim loops begin.
    ///
    /// Example, for Reduce, this creates accumulator state sized to the output brick.
    ///
    /// `access` is this node's iter→global access relation — used to derive the
    /// output coord that sizes the accumulator's initial state.
    Init { node: NodeName, access: AccessRelation },

    /// Per-iteration "compute" step. Examples:
    /// - Source: read brick at current coord into workspace.
    /// - Blockwise: read inputs from workspace, broadcast, apply op, store result.
    /// - Reduce: fold `workspace[input]` into accumulator.
    ///
    /// `inputs` names the workspace entries this step reads, in port order.
    /// Each input brick's coordinate was already fixed by the producer's own
    /// `Apply` (recompute model: the emission immediately precedes this read),
    /// so the edge carries no access of its own.
    /// The planner controls placement — a source Apply can be hoisted above
    /// inner loops for reuse when broadcasting is involved.
    ///
    /// `access` is this node's own iter→global translation (e.g. for the accumulator result).
    ///
    /// Sinks never appear in Apply — they have their own InitSink / WriteSink /
    /// FinalizeSink variants.
    Apply { node: NodeName, inputs: Vec<NodeName>, access: AccessRelation },

    /// Extract result from a compute node's state after core-dim iteration
    /// completes. Placed after the core-dim loops exit. Produces a brick
    /// in workspace for routing to the next step.
    ///
    /// For Reduce: calls accumulator.finalize() → workspace brick.
    Finalize { node: NodeName },

    /// FIXME: These sink-specific variants are a wart to me.
    /// Initialize a sink (allocate buffers, open files, etc.). Placed once,
    /// before any WriteSink into that sink executes.
    InitSink { sink: NodeName },

    /// Write `workspace[node]` into the sink at the current brick coord, and
    /// release `workspace[node]`. This is the explicit memory-release point
    /// for a brick on its way out of the system.
    ///
    /// `access` is this node's iter→global translation — used to derive the
    /// brick coord at which to write into the sink.
    WriteSink { node: NodeName, sink: NodeName, access: AccessRelation },

    /// Finalize a sink (commit writes, close files, etc.). Placed once,
    /// after every WriteSink into that sink has executed.
    FinalizeSink { sink: NodeName },
}

Binary Ops

A binary op (a + b) has to answer two questions at plan time: what is the result dtype, and which kernel runs? For primitives that’s just promotion. The interesting case is extension dtypes — datetime - datetime, minutes * i64 — where neither core nor the satellite alone has the whole picture. We resolve it with the same protocol Python uses for operator overloading: __op__ / __rop__.

Dispatch order

ArrayExpr::binary tries each leg in turn and takes the first that succeeds:

  1. Primitive promotion. Both operands primitive → promote to a common dtype (see Data Types) and build a two-input Binary node. This is a.__op__(b) for the built-in numeric types.
  2. The lhs’s capability (a.__op__(b)). If a’s dtype is an extension, ask its BinaryCapability to build the node.
  3. The rhs’s reflexive capability (b.__rop__(a)). If the op has a reflexive form, ask b’s capability with the operands swapped. Add/Mul are reflexive (commutative); Sub has none.
  4. Otherwise, error: ArrayOp::InvalidBinaryOpOperands — the analogue of Python’s TypeError after both __op__ and __rop__ decline.

The three-valued reply

Each capability leg returns an Option<Result<…>>, mirroring Python exactly:

replymeaningPython
None“not my op” — try the next legNotImplemented
Some(Ok(_))handled — here’s the built nodea returned value
Some(Err(_))“mine, but the operands are invalid”a raised error

The None-vs-Some(Err) distinction matters: a decline lets the protocol fall through to the reflexive leg, whereas an error stops it. So a capability declines structurally (wrong op, wrong other-dtype) and only errors once it has committed to handling the op.

What the kernel sees

Whichever leg wins, the node it produces is an ordinary Binary over a storage primitive — extension dtypes never reach a kernel. A capability gets there one of two ways (see Extensions): resolve its operands to their storage primitive and reuse core’s kernel (re-tagging the result), or build a node for its own satellite kernel.

So the runtime kernel is always monomorphic and primitive: dispatch_dtype! on the result tag picks one body, bounded on the Binary capability trait (add/sub/mul over a single T). Promotion guarantees both inputs already share that dtype, so the body never sees a mixed pair.

Indexing

We treat indexing operations as Op::Blockwise nodes with specialized access relations for inputs. The complexity in how we model access relations is purely to support indexing semantics.

At a high-level, we subset the input bricks appropriately, and then apply the appropriate per-brick indexer to construct the output. Below we work through three cases slicing, orthogonal indexing, and vectorized indexing.

Slicing

We need to represent three pieces of information:

  1. The output chunk grid, given the input grid.
  2. Knowledge of which indexers to apply to which chunk.

At runtime, the SliceKernel::clip receives an input brick, along with it’s (BrickCoordinate) in the input ChunkGrid, as well as the output BrickCoordinate in the output ChunkGrid.

The kernel embeds both chunk grids and slicers. Combined with the input & output coordinates, this is enough information to return an ArrayPayload of the right size for each input Brick.

Array layer

a.isel({"i": Indexer::Slicer(Slice::new(4, 12, 1))})

builds ArrayOp::Slice(HashMap<AxisName, Slice>) — one Slice { start, end, step } per indexed axis.

Chunk grid inference

Per sliced axis, the size of each output chunk is the intersection of one source chunk with [start, step, end). For now, we are clever and encode this as an output chunk grid that has AxisChunks::RLE along the subset axes when the source chunk grid has AxisChunks::Uniform.

Brix layer

If the start of the applied slice is on chunk-k of the input ChunkGrid, then output chunk 0 maps to input chunk-k. This information is recorded on the edge using

AxisAccess::Translated(
    TranslatedAccess::Independent(
        AxisOverride::Offset(k)
    )
)

for every sliced axis. That mouthful reads:

the output brick coordinate is translated to the input brick coordinate at runtime by applying an offset of k to the axis independent of the other axes participating in the Op.

Runtime (per-brick)

SliceKernel::clip constructs the output Brick by combining input and output chunk grids, with input and output brick coordinates together with the stored slicer.

  • Slicing an AxisChunks::Variable axis.

Reductions

A reduction applies a fold along one or more axes. Every reduction — whole-axis mean, coarsen, and the rest — is the same ArrayOp::Reduce node:

  • an accumulator says what to fold (sum, mean, …), and
  • a per-axis Frame says over which frames to fold it, in element space.

Only whole-axis and block window (coarsen) reductions are implemented. For these the access relation is AxisAccess::Composed: the frame becomes loop restructure applied at plan time rather than a runtime translation of the brick coordinates. (Note this approach cannot work for the general groupby Frame, but is fine for whole axis, block window, and sliding window aggregations).

Runtime (per-brick)

The accumulator folds each incoming brick’s frames into running state — for a whole-axis view the entire reduced extent is a single frame — and combines parallel states pairwise on merge. finalize divides sum / count (mean over integers promotes to DType::F64, NumPy-faithful) and squeezes out any axis whose view was keepdims: false. A Fill brick contributes value × volume without densifying.

Not yet implemented

  • Reducing along a sparse (occupancy-carrying) axis — hard error at lowering.
  • nanmean / skip-NA semantics.

Coarsen

Block-reduce an axis with fixed-width windows.

Array layer

a.coarsen({"i": 2}).sum()

builds ArrayOp::Reduce with a Frame::Window { size: 2, stride: 2 } view.

Chunk grid inference

Instead of dropping the viewed axis, its chunks are coarsened — the output extent along it becomes ceil(n / size).

Brix layer

Op::Reduce(LoweredAccumulator). The element-space window is translated to brick-space. There are two regimes (w = number of source bricks a window spans):

w == 1 — one or more windows fit inside one brick. Here the access relation on the edge is identity. All folding happens in element space inside the accumulator, invisible to the planner. There is no cross-brick communication.

w > 1 — the window spans multiple bricks. The access relation is:

AxisAccess::Composed(Frame::Window { size: w, stride: w })

which reads as

the planner builds a fold loop over each window of w bricks.

Planner

Constructing the fold space is the interesting part. The planner takes the coarsen axis of size n at its input extent and NamedLayout::logical_divides it by the window w, which splits the single axis into a two-mode (sub-axis) layout following (n : s).logical_divide(w) = (ceil(n/w), w) : (w·s, s).

for c in 0..ceil(n/w):           # mode 0, stride w·s — surviving output chunk
    for j in 0..w: # mode 1, stride s    — window lane, clipped on the last block
        process input brick i = c·(w·s) + j·s   # original brick index

This split is just a nested loop over the one input physical axis, represented through the CuTe layout algebra we use to model IterationSpace.

Schedule::Loop statements are emitted for each mode_index to construct the nested loop:

Domain  i ↦ (4, 2):(2, 1)
└─ Loop { axis: i, mode_index: 0 }      # c ∈ 0..4  — surviving output chunk; loop over windows
   ├─ Init     Reduce                    #   allocate accumulator for output coord c
   ├─ Loop { axis: i, mode_index: 1 }    # j ∈ 0..2  — fold each window
   │  ├─ Apply Source                    #     read input chunk i = 2·c + j
   │  └─ Apply Reduce ← [Source]         #     fold that chunk into accumulator[c]
   └─ Finalize Reduce                    #   finalize() → output chunk c

When n isn’t divisible by w, logical_divide rounds the block count up via ceil, so the last window is smaller. The NamedLayout cannot express this fact, so we add a sidecar IterationSpace::limits to limit the inner loop to the right extent (see Limit override).

Runtime (per-brick)

The runtime machinery is identical across all reductions (see Folding over frames). Each brick is cut into Window frames and folded once per (output lane, frame), so a window that lives inside one brick folds here rather than in the loop nest.

Not yet implemented

  • Sliding windows (stride != size) — hard error.
  • The other Frame variants (Blocks, PeriodicBlocks, VariableWindow, Groups) don’t lower yet.
  • Reducing along a sparse axis — same restriction as above.

Pad

  • Pad modifies first and last blocks; conceptually similar to slicing.
  • Two ways; we modify the brick “in-place” i.e. number of bricks does not change
    • we add new smaller bricks on either end.

Overlap

  • Overlapping views are stride≠1 on the access relation.
  • How do we (avoid) doing comms?
    • we could always recompute input bricks.
    • in this view we globally partition the strided iteration space between workers and each worker just recomputes what it needs. This removes the need to implement an exchange operator early on.
    • requires optimization where we push striding on the access relation through blockwise as close to the source as possible.
